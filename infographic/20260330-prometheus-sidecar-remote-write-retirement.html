📋 概要

Azure Monitor は、Prometheus メトリクスを Azure Monitor ワークスペースにリモートライトするためのサイドカーコンテナー (prom-remotewrite) を 2027 年 3 月 31 日に廃止することを発表した。これは、信頼性の向上と複雑性の削減を目的とした取り組みの一環であり、Prometheus のネイティブ remote_write 機能への移行が推奨される。

⚠ Before (廃止予定)

  • サイドカーコンテナー (prom-remotewrite) を各 Prometheus Pod にデプロイ・管理する必要があった
  • サイドカーの設定 (INGESTION_URL、IDENTITY_TYPE、AZURE_CLIENT_ID など) を個別に管理する必要があった
  • サイドカーコンテナーのリソース消費やリスタートなどの追加的な運用負荷が発生していた

✔ After (推奨)

  • Prometheus のネイティブ remote_write 設定で直接 Azure Monitor ワークスペースに送信可能
  • サイドカーコンテナーが不要になり、Pod 構成がシンプルになる
  • Prometheus の標準的な設定ファイル (prometheus.yml) のみで構成を管理できる

⭐ 主要な変更点

サイドカーコンテナーの廃止

prom-remotewrite サイドカーコンテナーが 2027 年 3 月 31 日に廃止される。各 Prometheus Pod へのサイドカーデプロイが不要になり、Pod 構成が大幅にシンプルになる。

ネイティブ remote_write への移行

Prometheus 標準の remote_write 設定で直接 Azure Monitor ワークスペースに送信可能。prometheus.yml の設定のみで構成を管理でき、追加コンポーネントが不要。

Azure AD ネイティブ認証サポート

Prometheus のネイティブ Azure AD 認証を利用し、システム割り当てマネージド ID、ユーザー割り当てマネージド ID、Microsoft Entra ID、ワークロード ID の 4 方式をサポート。

運用負荷の大幅削減

サイドカーコンテナーのイメージ更新、リソース管理、障害対応が不要になり、トラブルシューティングも容易になる。管理すべきコンポーネントが減少。

📝 設定方法

Prometheus 設定ファイルの更新 (マネージド ID 使用時)

remote_write:
  - url: "<Azure Monitor ワークスペースのメトリクス取り込みエンドポイント>"
    azuread:
      cloud: 'AzurePublic'
      managed_identity:
        client_id: "<マネージド ID のクライアント ID>"

Prometheus Operator 使用時

prometheus:
  prometheusSpec:
    remoteWrite:
    - url: "<Azure Monitor ワークスペースのメトリクス取り込みエンドポイント>"
      azureAd:
        cloud: 'AzurePublic'
        managedIdentity:
          clientId: "<マネージド ID のクライアント ID>"

✅ メリット

技術面

  • サイドカーコンテナーが不要になり、Pod のリソース消費が削減される
  • Prometheus の標準機能のみで構成できるため、構成がシンプルになる
  • サイドカーコンテナーの障害やリスタートに起因する問題が解消される

運用面

  • 管理すべきコンポーネントが減少し、運用負荷が軽減される
  • サイドカーコンテナーのイメージ更新が不要になる
  • トラブルシューティングが容易になる

⚠ 制約事項・注意点

  • 既存環境では 2027 年 3 月 31 日までに移行作業が必要
  • Prometheus のバージョンアップが必要な場合がある (最小 2.45 以上)
  • ロール割り当て反映まで最大 30 分かかる場合がある (HTTP 403 エラーの可能性)

💡 ユースケース

1

AKS クラスターからの移行

シナリオ: サイドカーを使用中の AKS 環境で、prom-remotewrite コンテナーを利用して Azure Monitor ワークスペースにメトリクスを送信している。
効果: ネイティブ remote_write への移行により、サイドカーの管理コストが不要になり、Pod 構成がシンプルになる。Prometheus の設定ファイルのみで構成を管理でき、運用負荷が大幅に軽減される。
2

マルチクラスター環境での集約

シナリオ: 複数のセルフホスト Prometheus クラスターから 1 つの Azure Monitor ワークスペースへメトリクスを集約したい。
効果: 各クラスターの Prometheus 設定ファイルに remote_write を追加するだけで、サイドカーのデプロイなしに統一的なメトリクス集約が実現できる。構成の一貫性が保たれ、大規模環境での管理が容易になる。
