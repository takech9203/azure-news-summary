Azure Monitor In Preview

AKS アプリケーションの OpenTelemetry モニタリングがパブリックプレビュー開始

2026-04-13 | AKS + OpenTelemetry Protocol (OTLP) + Application Insights
📋

概要

Azure Monitor が AKS 上のアプリケーション監視において、OpenTelemetry Protocol (OTLP) を使用した計装とデータ収集のパブリックプレビューを開始。コード変更なしに自動計装でトレース・メトリクス・ログを収集できるようになった。

  • AKS ワークロードに Azure Monitor OpenTelemetry ディストリビューションの自動計装 (autoinstrumentation) をデプロイ可能
  • ネームスペースまたはデプロイメント単位でモニタリングを有効化 -- コード変更不要
  • 既に OpenTelemetry SDK で計装済みのアプリも、自動構成 (autoconfiguration) で Application Insights にエクスポート
  • Application Insights と Container Insights の統合により、アプリとインフラの両レベルで統合監視

Before

  • 各アプリに手動で OpenTelemetry SDK を組み込み、接続文字列やエクスポーターをコード内で管理
  • マイクロサービスごとに計装コードを追加・保守する高負荷。多言語環境では対応困難
  • アプリテレメトリ (トレース/メトリクス) とインフラ監視 (Container Insights) が別管理
  • OTel シグナルの複数宛先ルーティング用コレクター構成を自前管理

After

  • ネームスペース/デプロイメント単位で自動計装を有効化、コード変更なしにテレメトリ収集開始
  • OTel ディストリビューションが Pod に自動注入、計装管理を AKS プラットフォーム側に委譲
  • Application Insights + Container Insights 統合で、アプリとインフラをコンテキスト付きで分析
  • 計装済みアプリも自動構成で環境変数設定のみで Application Insights にエクスポート可能

主要機能

🚀

1. 自動計装 (Autoinstrumentation)

Azure Monitor OpenTelemetry ディストリビューションを AKS の Pod に自動注入。Java および Node.js アプリケーションに対応し、コード変更なしにトレース、メトリクス、ログの収集を開始できる。ネームスペース単位で言語を指定するか、デプロイメントごとにアノテーションで個別に制御可能。

⚙

2. 自動構成 (Autoconfiguration)

既に OpenTelemetry SDK で計装済みのアプリケーションに対して、環境変数を設定することで Application Insights へのテレメトリエクスポートを有効化。アプリケーションコードの変更は不要で、プラットフォームレベルで OTLP エクスポーターの宛先を設定する。

📡

3. OTLP インジェスト対応の Application Insights

Application Insights リソースで OTLP サポートを有効化し、OpenTelemetry Protocol による直接的なテレメトリ取り込みが可能に。マネージドワークスペースと組み合わせて使用する。

🔗

4. Container Insights との統合

Application Insights のテレメトリと Container Insights のインフラストラクチャ情報を統合。コントローラービューからリクエスト障害、遅延オペレーション、推奨される調査を確認でき、Application Map からドリルダウンして Pod の状態を確認することも可能。

☸

5. Kubernetes カスタムリソースによるオンボーディング

Instrumentation カスタムリソースを使用して、ネームスペース単位またはデプロイメント単位で監視を構成。OpenTelemetry のアノテーション (instrumentation.opentelemetry.io/inject-java など) でデプロイメントごとの設定をオーバーライド可能。

設定方法

Azure CLI

1

aks-preview 拡張機能のインストール

bash
# aks-preview 拡張機能のインストール
az extension add --name aks-preview
az extension update --name aks-preview
2

プレビュー機能フラグの登録

bash
# プレビュー機能フラグの登録 (AKS サブスクリプション)
az feature register --namespace "Microsoft.ContainerService" \
  --name "AzureMonitorAppMonitoringPreview"

# 登録状態の確認 (Registered になるまで待機)
az feature list -o table --query \
  "[?contains(name, 'Microsoft.ContainerService/AzureMonitorAppMonitoringPreview')].{Name:name,State:properties.state}"

# リソースプロバイダーの再登録
az provider register --namespace "Microsoft.ContainerService"
3

Application Insights OTLP プレビュー機能フラグの登録

bash
# Application Insights OTLP プレビュー機能フラグの登録
az feature register --name OtlpApplicationInsights \
  --namespace Microsoft.Insights
az provider register -n Microsoft.Insights
4

既存クラスターでアプリケーション監視を有効化

bash
# 既存クラスターでアプリケーション監視を有効化
az aks update \
  --resource-group <resource-group> \
  --name <cluster-name> \
  --enable-azure-monitor-app-monitoring

Azure Portal

1

クラスターの準備

  1. AKS リソースの「Monitor」ペインを開く
  2. 「Enable support for Autoinstrumentation」と「Enable support for data collection from vendor neutral OpenTelemetry SDKs (Preview)」を有効にする
  3. 「Review + enable」を選択
2

OTLP 対応の Application Insights リソースの作成

  1. Azure Portal で新しい Application Insights リソースを作成
  2. 「Enable OTLP Support (Preview)」を有効にする
  3. 「Use managed workspaces」を「Yes」に設定
  4. インフラストラクチャメトリクスに使用するワークスペースとは別の Azure Monitor ワークスペースを指定
3

アプリケーションのオンボーディング

  1. AKS リソースの「Kubernetes resources」>「Namespaces」から対象のネームスペースを選択
  2. 「Application Monitoring (Preview)」を選択
  3. OTLP 対応の Application Insights リソースを選択
  4. 計装タイプを選択 (Java 自動計装 / Node.js 自動計装 / ユーザー構成)
  5. 「Configure」を選択
4

デプロイメントの再起動

bash
kubectl rollout restart deployment -n <namespace>
5

計装状態の確認

「Application Monitoring (Preview)」に戻り、デプロイメントが「Instrumented」ステータスになっていることを確認

ユースケース

🏬

マイクロサービスの統合可観測性

数十の Java / Node.js マイクロサービスで構成される EC サイトの AKS クラスターに対して、ネームスペース単位で自動計装を有効化し、全サービスのテレメトリを Application Insights に集約。
効果: 個別 SDK 組み込み作業が不要に。Application Map でサービス間依存関係を可視化し、遅延が Pod リソース不足かコードの問題かを迅速に切り分け可能。
🔄

既存 OTel 計装の AppInsights 統合

既に OpenTelemetry SDK で計装し Jaeger / Prometheus にテレメトリ送信中のアプリに対して、自動構成で Application Insights への追加エクスポートを有効化。
効果: 既存計装コードを変更せず Azure Monitor の分析機能を利用可能。デプロイメントのアノテーション追加のみで完了。
🛠

開発環境での迅速なデバッグ

開発・ステージング環境の AKS クラスターで自動計装を有効化し、新機能デプロイ時にトレースとログを即座に確認できる環境を構築。
効果: モニタリングセットアップの時間をゼロに。デプロイ直後から分散トレースを確認してボトルネックやエラー原因を特定。
💰

料金情報

Application Insights の OTLP インジェストは、Azure Monitor の Application Insights 料金体系に基づいて課金される。サンプリング設定やログの詳細度を適切に構成することでコストを最適化可能。

⚠

制限事項・注意点

  • パブリックプレビューであり、SLA は提供されない。本番ワークロードでの利用は推奨されない
  • 自動計装の対応言語は現時点で Java と Node.js のみ (.NET と Python は限定プレビュー)
  • Windows ノードプールおよび Linux Arm64 ノードプールは非対応
  • OTLP/HTTP (バイナリ Protobuf) のみサポート。JSON ペイロードや OTLP/gRPC は非対応
  • OpenTelemetry SDK エクスポーターの圧縮はサポートされていない
  • Istio mTLS が有効なネームスペースでは動作しない
  • DCR アソシエーションは 1 クラスターあたり最大 30 まで
  • Israel Central、Israel North West、Qatar Central、UAE North、UAE Central リージョンでは利用不可
  • AKS HTTP プロキシ構成、Private Link 構成、デュアルスタッククラスターは未検証

関連サービス

Application Insights

AKS アプリケーションからのテレメトリの格納先。Application Map、障害分析、パフォーマンス分析、Live Metrics などのエクスペリエンスを提供。

Container Insights

AKS クラスターのインフラストラクチャ監視 (コンテナログ、Prometheus メトリクス) を提供。アプリテレメトリと統合して分析可能。

Azure Managed Grafana

Prometheus メトリクスのダッシュボード表示に使用。AKS クラスターの監視設定時に連携可能。

Azure Monitor OpenTelemetry Distro

.NET、Java、Node.js、Python 向けの OpenTelemetry ディストリビューション。AKS Pod に自動注入される計装ライブラリ。

Azure Monitor ワークスペース

OTLP 対応 Application Insights がメトリクスを格納するマネージドワークスペース。
