Azure Cosmos DB for PostgreSQL Database Retirement 2026-03-30

Azure Cosmos DB for PostgreSQL 廃止と Elastic Cluster への移行

Azure Cosmos DB for PostgreSQL は 2029 年 3 月 31 日に廃止されます。後継サービス Azure Database for PostgreSQL Elastic Cluster への移行が必要です。移行ツールは 2026 年 4 月 13 日から利用可能で、通常 10 分以内に完了します。

🔄 概要: Before / After 比較

Azure Cosmos DB for PostgreSQL は今後新機能の追加が限定的となり廃止パスに入っています。後継の Elastic Cluster は Flexible Server 基盤で継続的な投資が行われ、多くの面で機能が強化されています。

Before: Cosmos DB for PostgreSQL

  • 専用コーディネーターノード（追加課金あり）
  • クエリアクセスがコーディネーターに限定
  • PostgreSQL 17/18 非対応
  • Managed Identity 非対応
  • PgBouncer 非対応
  • 最大接続数: 2,500/ノード

After: Elastic Cluster

  • 専用コーディネーター不要（コスト削減）
  • 任意のノードからクエリ実行可能
  • PostgreSQL 17/18 対応
  • Managed Identity / Entra ID 認証
  • PgBouncer サポート
  • 最大接続数: 3,000/ノード
重要: 2029 年 3 月 31 日以降、Azure Cosmos DB for PostgreSQL は利用できなくなります。早期に移行計画を策定し、廃止日までに移行を完了してください。

✨ Elastic Cluster の主な優位点
💰

専用コーディネーター不要

別途課金されるコーディネーターノードが不要となり、ベースラインコストを削減。コスト構造がシンプルになります。

🔀

任意ノードからのクエリ実行

すべてのノードからクエリアクセスが可能で、運用の柔軟性が向上。ツーリングやトラブルシューティングも容易になります。

🐘

最新 PostgreSQL バージョン対応

PostgreSQL 17/18 をサポート。最新のセキュリティ更新やパフォーマンス改善を早期に活用できます。

🔐

Managed Identity / Microsoft Entra ID 認証

エンタープライズ ID 管理との統合が強化され、シークレット管理が簡素化されます。

🔗

PgBouncer サポート

接続プーリングが利用可能。ノードあたり最大 3,000 接続（従来は最大 2,500）に向上しています。

🏢

Flexible Server 基盤

バックアップ、監視、メンテナンス制御など Flexible Server の成熟した運用モデルを継承。継続的な機能投資が行われます。

🚀 移行手順
1

Azure Portal でクラスターページを開く

Azure Cosmos DB for PostgreSQL クラスターページにアクセスし、「移行」タブを選択します。

2

事前検証チェック

ポータルが自動的に事前検証を実行します。拡張機能や設定の互換性がチェックされます。

3

ソースを読み取り専用に切替

移行中、ソースクラスターが読み取り専用に切り替わります（書き込みロック: 平均 5-8 分）。

4

スナップショット作成・データコピー

スナップショットが作成され、ターゲット Elastic Cluster がプロビジョニングされます。差分ファイル（拡張機能、設定、証明書、WAL ログ等）もコピーされます。

5

移行完了・接続文字列の更新

ユーザー設定（カスタム設定、HA 設定）が適用され、移行完了。新しい接続文字列に切り替えてください。Private Endpoint は再作成が必要です。

移行時間の目安: エンドツーエンドの移行は通常 10 分以内 に完了。書き込みロック（読み取り専用）ウィンドウは平均 5-8 分 程度。標準のメンテナンスウィンドウ内で実行可能です。

⚠️ 制約事項・注意点
⏱️

書き込みロック

移行中は一時的に書き込みロック（読み取り専用）ウィンドウが発生します（平均 5-8 分）。

🔌

Private Endpoint の再作成

Private Endpoint を使用している場合は移行後に再作成が必要です。

📦

一部拡張機能の非対応

クラスターモードでの TimescaleDB など、一部の拡張機能はサポートされない場合があります。

🔧

GA 後対応予定の機能

計画的フェイルオーバー、Storage autogrow、Premium SSD v2（80K IOPS/ノード）は GA 後に対応予定です。

🌐

Virtual Network

Virtual Network は両サービスとも非対応です。ノードの削除はリバランスによるデータ移動のみで、自動デプロビジョニングは行われません。
