(Linux Consumption) Retirement

Linux Consumption プラン上のランタイム v3 アプリケーションの実行停止

Retirement 2026-04-17

Overview

Action Required: 2026-09-30 Deadline

Linux Consumption プラン上で動作する Azure Functions ランタイム v3 のアプリケーションは 2026 年 9 月 30 日 以降に実行が完全に停止します。期限までに v4 ランタイムへの移行が必須です。

Microsoft Azure は、Linux Consumption プラン上で動作する Azure Functions ランタイム v3 のアプリケーションが 2026 年 9 月 30 日以降に実行を停止することを発表した。ランタイム v3 は 2022 年 12 月 13 日に既に正式に廃止 (End of Life) となっているが、レガシーインフラストラクチャへの依存を削減しサポート対象のプラットフォームへの投資を集中するため、v3 ランタイムの強制的な廃止が実施される。Azure Functions v4 は v3 との後方互換性が高く、多くのアプリケーションでは大きなコード変更なしに移行が可能である。

Key Points

  • Linux Consumption プラン上のランタイム v3 Function App が 2026/9/30 に実行停止
  • ランタイム v3 は 2022/12/13 に既に正式廃止済み (セキュリティパッチ・バグ修正なし)
  • v4 ランタイムへの移行は後方互換性が高く、多くの場合は少ないコード変更で対応可能
  • Linux Consumption プラン自体も 2028/9/30 に廃止予定 (Flex Consumption へ移行推奨)
  • Windows Consumption プランのアプリケーションは現時点では影響を受けない

Before (v3 Runtime)

  • 2022/12/13 に正式廃止済みだが Linux Consumption 上で動作継続
  • セキュリティパッチやバグ修正を受けていない状態
  • レガシーインフラストラクチャへの依存が残存
  • 古い言語バージョン (.NET Core 3.1, Node.js 12 等) に制約
  • Azure Functions Proxies などの廃止予定機能に依存

After (v4 Runtime)

  • サポート対象プラットフォームで SLA の適用を維持
  • セキュリティパッチ・バグ修正が継続的に提供
  • 最新言語バージョン (.NET 8/10, Node.js 20/22, Python 3.12) を利用可能
  • C# isolated worker model で将来の .NET 対応が容易に
  • 強化されたモニタリング・Key Vault 統合などの新機能を活用

Breaking Changes (v3 to v4)

⛔

Azure Functions Proxies

v4 では非サポート。API Management への移行が必要。Proxies 機能を使用して API ルーティングやレスポンス変換を行っていた場合、Azure API Management で同等の機能を構成する。

⛔

AzureWebJobsDashboard

v4 では非サポート。Application Insights を使用したモニタリングに移行する。AzureWebJobsDashboard アプリケーション設定は削除し、APPINSIGHTS_INSTRUMENTATIONKEY を設定。

🔒

Extension Minimum Version

v4 ではバインディング拡張機能の最小バージョン要件が強制される。extension bundle を 2.x 以上に更新し、すべてのバインディングパッケージが最新バージョンであることを確認する。

🔑

Key Vault Provider

Key Vault 参照プロバイダーが Azure.Identity / Azure.Security.KeyVault.Secrets に変更。古い Key Vault SDK に依存していたコードは更新が必要。

⏱

Linux Consumption Timeout

v4 ではデフォルトおよび最大タイムアウトが強制される。長時間実行されるファンクションではタイムアウト設定を見直し、必要に応じて処理の分割を検討する。

📛

Host ID Uniqueness

ストレージアカウントを共有するアプリではホスト ID が一意である必要がある。同一ストレージアカウントを使用する複数の Function App がある場合、ホスト ID の重複がないことを確認する。

Use Cases

UC1: IoT Data Pipeline Migration

Linux Consumption プラン上で Azure Functions v3 (Python 3.8) を使用して IoT Hub からのイベントを処理し、Cosmos DB に格納するパイプラインを運用している組織が、v4 ランタイムへ移行する。

Migration Steps:
  • extension bundle を 2.x 以上に更新
  • Python バージョンを 3.10 以上に更新
  • FUNCTIONS_EXTENSION_VERSION を ~4 に変更
  • linuxFxVersion を Python|3.10 に更新

-- 最新の Python ランタイムとバインディングにより、パフォーマンスの向上とセキュリティの強化が期待できる。

UC2: .NET Web API Modernization

Linux Consumption プラン上で Azure Functions v3 (C# / .NET Core 3.1 in-process) を使用して REST API を提供している組織が、v4 ランタイムおよび .NET 8 isolated worker model へ移行する。

Migration Steps:
  • .csproj の TargetFramework を net8.0、AzureFunctionsVersion を v4 に変更
  • Microsoft.NET.Sdk.Functions パッケージを Microsoft.Azure.Functions.Worker 系パッケージに置換
  • Program.cs を追加し、ConfigureFunctionsWebApplication() で ASP.NET Core 統合を構成
  • FUNCTIONS_WORKER_RUNTIME を dotnet-isolated に変更

-- ASP.NET Core 統合によるパフォーマンス向上と、isolated worker model による将来の .NET バージョンへのスムーズな対応が可能になる。

Code Samples

Find Affected Apps (PowerShell)

PowerShell 
$Subscription = '<YOUR SUBSCRIPTION ID>'

Set-AzContext -Subscription $Subscription | Out-Null

$FunctionApps = Get-AzFunctionApp

$AppInfo = @{}

foreach ($App in $FunctionApps)
{
     if ($App.ApplicationSettings["FUNCTIONS_EXTENSION_VERSION"] -like '*3*')
     {
          $AppInfo.Add($App.Name, $App.ApplicationSettings["FUNCTIONS_EXTENSION_VERSION"])
     }
}

$AppInfo

Update Runtime Version (Azure CLI)

Bash 
# Check current runtime version
az functionapp config appsettings list \
  --name <APP_NAME> \
  --resource-group <RESOURCE_GROUP_NAME> \
  --query "[?name=='FUNCTIONS_EXTENSION_VERSION']"

# Update runtime version to v4
az functionapp config appsettings set \
  --settings FUNCTIONS_EXTENSION_VERSION=~4 \
  -g <RESOURCE_GROUP_NAME> \
  -n <APP_NAME>

# Update linuxFxVersion (e.g., Python 3.10)
az functionapp config set \
  --name <APP_NAME> \
  --resource-group <RESOURCE_GROUP_NAME> \
  --linux-fx-version "Python|3.10"

Safe Migration with Staging Slot

Bash 
# Set WEBSITE_OVERRIDE_STICKY_EXTENSION_VERSIONS on staging slot
az functionapp config appsettings set \
  --settings WEBSITE_OVERRIDE_STICKY_EXTENSION_VERSIONS=0 \
  -g <RESOURCE_GROUP_NAME> \
  -n <APP_NAME> \
  --slot staging

# Update runtime version on staging slot
az functionapp config appsettings set \
  --settings FUNCTIONS_EXTENSION_VERSION=~4 \
  -g <RESOURCE_GROUP_NAME> \
  -n <APP_NAME> \
  --slot staging

# Swap staging to production after verification
az functionapp deployment slot swap \
  -g <RESOURCE_GROUP_NAME> \
  -n <APP_NAME> \
  --slot staging \
  --target-slot production

Benefits of Migration

Business

  • サポート対象プラットフォーム上で SLA の適用を維持し、サービス継続を確保
  • セキュリティパッチ・バグ修正が提供される最新ランタイムでコンプライアンスリスクを低減

Technical

  • .NET 8/10、Node.js 20/22、Python 3.12 など最新言語バージョンを利用可能
  • C# isolated worker model で将来の .NET バージョンへの対応が容易に
  • extension bundle 最新バージョンで新しいバインディング機能やパフォーマンス改善を活用
  • 強化されたモニタリング・Key Vault 統合などの v4 プラットフォーム機能を利用可能

Limitations / Caveats

  • Azure Functions Proxies は v4 で利用不可。API Management への移行が必要
  • 一部バインディングのパッケージ名・クラス名が変更されており、コード修正が必要な場合がある
  • C# in-process model は .NET 8 までのサポート (2026/11/10 サポート終了)。isolated worker model への二段階移行が必要になる可能性
  • 移行後のアプリケーション動作検証に一定のテスト工数が必要
  • Linux Consumption プラン自体も 2028/9/30 に廃止予定。長期的には Flex Consumption への移行計画が必要
