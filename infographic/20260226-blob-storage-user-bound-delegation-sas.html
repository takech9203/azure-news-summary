User-Bound User Delegation SAS

SAS トークンの使用を特定の Entra ID ユーザーに制限し、トークン漏洩時のリスクを大幅に低減

Release: 2026-02-26 Service: Azure Blob Storage API: sv=2025-07-05+

📋 Overview

Azure Storage に新たに導入された「ユーザーバウンド ユーザー委任 SAS」は、従来のユーザー委任 SAS のセキュリティをさらに強化する機能である。SAS トークン作成時にエンドユーザーの Entra ID オブジェクト ID (sduoid) を指定し、使用時には Entra ID の Bearer トークンによる本人確認を必須とすることで、指定されたユーザー以外は SAS トークンを使用できない。

👤

Bearer トークンによる本人確認で、不正利用を確実に防止

🌐

クロステナント対応により、外部パートナーへの安全なデータ共有が可能

🛡

ゼロトラストアーキテクチャとの親和性が高く、多層的なセキュリティを実現

Before

  • SAS トークンを取得した任意のクライアントがリソースにアクセス可能
  • トークン漏洩時、有効期限内は誰でもアクセスできた
  • マルチユーザー環境で使用者を特定ユーザーに限定する手段がなかった
  • 使用者制御にはユーザー委任キーの無効化（最大 7 日待ち）しか手段がなかった

After

  • SAS トークンの使用を特定の Entra ID ユーザーに限定可能
  • Bearer トークンによる本人確認が必須となり不正利用を防止
  • クロステナントシナリオにも対応し、異なるテナントのユーザーへの SAS 発行が可能
  • SAS の柔軟性と Entra ID のユーザーバウンドアクセスを両立

ユーザーバウンド ユーザー委任 SAS の認証フロー。アプリケーションが Entra ID を通じてユーザー委任キーを取得し、特定ユーザーにバウンドされた SAS トークンを生成する。エンドユーザーは SAS トークンに加えて Entra ID の Bearer トークンを提示する必要があり、Azure Storage は oid クレームと sduoid パラメータを照合して本人確認を行う。攻撃者が SAS トークンを入手しても、対応する Entra ID 認証情報がなければアクセスは拒否される。

⚙ Key Features

🔒

ユーザーバウンド SAS トークン (sduoid)

SAS トークン作成時に signedDelegatedUserObjectId (sduoid) フィールドでエンドユーザーの Entra ID オブジェクト ID を指定する。

  • 使用時にユーザーは Entra ID の Bearer トークンを提示する必要がある
  • Azure Storage は Bearer トークンの oid クレームと SAS の sduoid 値を照合
  • 一致しない場合はアクセスを拒否し、トークン漏洩時の不正利用を防止
🌐

クロステナント対応 (skdutid)

signedKeyDelegatedUserTenantId (skdutid) フィールドにより、異なるテナントのユーザーへの SAS 発行が可能。

  • Bearer トークンの tid クレームと skdutid 値が照合される
  • デフォルトで無効、allowCrossTenantDelegationSas プロパティで有効化
  • パートナー企業やゲストユーザーへの安全なデータ共有を実現
🔑

ユーザー委任キーの拡張

Get User Delegation Key API に DelegatedUserTid パラメータが追加された。

  • クロステナントシナリオで委任対象ユーザーのテナント ID を指定可能
  • レスポンスに SignedDelegatedUserTid フィールドが追加

2. ユーザーバウンド SAS トークンの使用

HTTP 
GET https://{account}.blob.core.windows.net/{container}/{blob}?{sas-token}
Authorization: Bearer {end-user-entra-token}

Azure Storage は Bearer トークン内の oid クレームが SAS の sduoid と一致するか検証する。SAS トークンパラメータとして sduoid、skdutid（クロステナントの場合）、sv=2025-07-05 以降を指定する。

💡 Use Cases

1

マルチユーザークラスタ環境でのデータアクセス

Hadoop や Spark クラスタにおいて、各ユーザーに個別のユーザーバウンド SAS を発行することで、共有クラスタ上でもユーザー単位のアクセス制御を実現する。

✅ 共有環境でも個々のユーザーの Blob アクセスを厳密に分離でき、データの機密性を確保。
2

外部パートナーへの一時的なデータ共有

パートナー企業の特定ユーザーに対してクロステナント対応のユーザーバウンド SAS を発行し、安全にデータを共有する。万が一 SAS トークンが漏洩しても、指定されたユーザー以外はアクセスできない。

✅ トークン漏洩時の被害範囲を限定し、クロステナントでの安全なコラボレーションを実現。
3

規制産業でのデータアクセス管理

金融・医療などの規制産業において、データアクセスを個人レベルで追跡・監査する必要がある場合に、ユーザーバウンド SAS によりアクセスの非否認性を確保する。

✅ コンプライアンス要件を満たす個人レベルのアクセス追跡・監査が可能。
4

SaaS アプリケーションでのテナント分離

SaaS プロバイダーがエンドユーザーに Blob Storage への直接アクセスを提供する際に、ユーザーバウンド SAS により各ユーザーのアクセスを厳密に分離する。

✅ マルチテナント環境でもユーザー単位のアクセス分離を実現し、データ漏洩リスクを最小化。

⚠ Limitations / Caveats

⚠️ パブリックプレビュー段階 - 本番環境での使用は推奨されない。GA 時に仕様変更が発生する可能性がある。
⚠️ 追加の認証オーバーヘッド - SAS トークンに加えて Entra ID の Bearer トークンも必要となるため、クライアント側の実装が複雑になる。
⚠️ API バージョン要件 - sv=2025-07-05 以降が必要。古い SDK やツールでは利用できない場合がある。
⚠️ クロステナントのデフォルト無効 - ストレージアカウントレベルでの allowCrossTenantDelegationSas の明示的な設定変更が必要。
⚠️ ストアドアクセスポリシー非対応 - アドホック SAS としてのみ使用可能。
⚠️ ユーザー委任キーの有効期間制限 - 最大 7 日間。長期間のアクセス委任には定期的なキー更新が必要。
