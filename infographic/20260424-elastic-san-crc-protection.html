Azure Elastic SAN

CRC Protection (CRC-32C チェックサム検証) が一般提供開始

iSCSI ヘッダーとデータペイロードに CRC-32C チェックサムを適用し、サイレントデータ破損リスクを低減

Launched (GA) 2026-04-24

概要

Azure Elastic SAN の CRC Protection 機能が一般提供 (GA) となった。iSCSI プロトコルのヘッダーおよびデータペイロードに CRC-32C (Castagnoli) チェックサムを適用し、ネットワーク機器やメモリにおけるビットフリップに起因するサイレントデータ破損を検出・防止する。ボリュームグループ単位で適用され、グループ内の全ボリュームが設定を継承する。有効化すると CRC-32C 未設定のクライアント接続を拒否し、データ整合性を強制する。新規・既存の Elastic SAN いずれにも設定可能で、追加料金は不要。

Key Points

  • CRC-32C チェックサムを iSCSI ヘッダーとデータダイジェストの両方に適用
  • ボリュームグループ単位で設定 -- グループ内の全ボリュームが継承
  • 有効化時、CRC-32C 未設定のクライアント接続を拒否
  • 新規・既存の Elastic SAN いずれにも設定可能
  • 追加料金なし -- Elastic SAN の標準機能として提供

Before

  • データ整合性が TCP チェックサムのみに依存し、サイレントデータ破損の検出能力が限定的
  • CRC-32C 未設定のクライアントも接続を許可されていた
  • ネットワーク機器やメモリにおけるビットフリップに対する保護が不十分

After

  • CRC-32C チェックサムを iSCSI ヘッダーとデータペイロードの両方に適用
  • ボリュームグループ単位の強制により CRC-32C 未設定クライアントの接続を拒否
  • 新規・既存の Elastic SAN いずれにも適用可能
  • サイレントデータ破損リスクを大幅に低減

🛡

CRC-32C チェックサム検証 (iSCSI ヘッダー + データダイジェスト)

iSCSI プロトコルのヘッダーダイジェストとデータダイジェストの両方に CRC-32C (Castagnoli) チェックサムを適用。ネットワーク転送中のビットフリップやサイレントデータ破損を検出し、データ整合性を保証する。

📁

ボリュームグループ単位の強制 -- 全ボリュームが継承

CRC Protection はボリュームグループレベルで設定され、グループ内のすべてのボリュームが自動的に設定を継承する。個別のボリュームごとの設定は不要で、一貫した保護を実現。

🚫

接続制御 -- CRC-32C 未設定クライアントの拒否

CRC Protection を有効化すると、CRC-32C チェックサムを設定していないクライアントからの接続を拒否する。これにより、保護されていない接続によるデータ破損リスクを排除する。

⚙

既存環境サポート -- 既存 SAN でも有効化可能

新規作成時だけでなく、既存の Elastic SAN のボリュームグループに対しても後から CRC Protection を有効化できる。段階的な導入が可能。

🔄

柔軟な無効化 -- 無効時は接続を拒否しない

CRC Protection を無効に設定した場合、CRC-32C 未設定のクライアントからの接続も許可される。互換性を優先する場合は無効のままで運用可能。

Code Samples

# PowerShell - Create volume group with CRC Protection
New-AzElasticSanVolumeGroup `
  -ResourceGroupName $RgName `
  -ElasticSANName $EsanName `
  -Name $EsanVgName `
  -EnforceDataIntegrityCheckForIscsi $true
# Azure CLI - Create volume group with CRC Protection
az elastic-san volume-group create \
  --elastic-san-name $EsanName \
  -g $RgName \
  -n $EsanVgName \
  --data-integrity-check true

Use Cases

UC1: ミッションクリティカルなデータベース (SQL Server, Oracle)

金融・医療など高いデータ整合性が要求されるデータベースワークロードにおいて、ネットワーク転送中のサイレントデータ破損はデータ不整合やシステム障害の原因となる。CRC Protection により iSCSI レベルでの整合性検証が追加され、データベースの信頼性を向上させる。

-- iSCSI レベルでの CRC-32C 検証により、データベースの整合性保証を強化。

UC2: 金融・医療業界のワークロード

規制要件により高いデータ整合性が求められる金融・医療業界では、サイレントデータ破損の検出能力が重要なコンプライアンス要素となる。CRC Protection を有効化することで、ストレージレベルでのデータ整合性検証を追加し、規制要件への適合を支援する。

-- ストレージレベルの整合性検証により、規制コンプライアンス要件への適合を支援。

UC3: 大規模 AKS クラスターのストレージ

多数のノードから同時にアクセスされる大規模 AKS クラスター環境では、ネットワーク経路が複雑化しビットフリップのリスクが増加する。CRC Protection により、コンテナワークロードのストレージアクセスにおけるデータ整合性を確保する。

-- 大規模コンテナ環境でのストレージアクセスにおけるデータ整合性を確保。

UC4: ディザスタリカバリ環境

DR 環境では、バックアップデータの整合性が復旧時の成否を左右する。CRC Protection により、DR ストレージへの書き込み時にもデータ整合性を検証し、復旧時のデータ破損リスクを低減する。

-- DR ストレージのデータ整合性を保証し、確実な復旧を支援。

Limitations / Caveats

  • North Europe および South Central US リージョンでは利用不可
  • Fedora および下流の Linux ディストリビューション (RHEL, CentOS, Rocky) は iSCSI データダイジェストをサポートしていないため、有効化すると接続が失敗する
  • Azure VMware Solution との互換性なし
  • クライアント側の iSCSI イニシエーター設定が必要
  • 既存のボリュームグループで有効化すると、CRC-32C 未設定の既存クライアント接続が拒否される可能性がある
