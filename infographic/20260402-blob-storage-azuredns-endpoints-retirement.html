Azure Blob Storage Retirement

AzureDnsZone エンドポイント (プレビュー) の廃止予告

2026-04-02 | Storage カテゴリ | 廃止期限: 2027 年 3 月

📋 概要

Azure Blob Storage アカウントの AzureDnsZone エンドポイント (プレビュー) が、2027 年 3 月をもって廃止されることが発表された。現在 AzureDnsZone エンドポイントを使用しているユーザーは、すべての新規および既存の Blob Storage アカウントのデプロイにおいて、Standard エンドポイントへの移行を行う必要がある。

AzureDnsZone エンドポイントは、1 サブスクリプション・1 リージョンあたり最大 5,000 のストレージアカウントを作成可能にするプレビュー機能として提供されていた。Standard エンドポイントでは既定で 250 (クォータ増加申請で最大 500) のストレージアカウントが作成可能である。

廃止の背景: AzureDnsZone エンドポイントはプレビュー段階にとどまり GA には至らなかった。また、エンドポイント URL にゾーン識別子 (z00-z50) を含むため、アプリケーション側で実行時にエンドポイントを動的に取得する必要があるという課題があった。

移行前後の比較

Before (AzureDnsZone)

  • URL 形式: account.z[XX].blob.storage.azure.net
  • ゾーン識別子 (z00-z50) が URL に含まれる
  • 実行時に Get Properties API でエンドポイントを動的取得が必要
  • プレビュー段階 (GA 未到達)
  • 最大 5,000 アカウント/リージョン

After (Standard)

  • URL 形式: account.blob.core.windows.net
  • アカウント名から直接 URL 構築可能
  • 実行時のエンドポイント問い合わせ不要
  • GA 済み (安定稼働)
  • 最大 250-500 アカウント/リージョン

🔑 移行の主要ポイント

🔗

エンドポイント URL の変更

account.z[00-50].blob.storage.azure.net 形式から account.blob.core.windows.net 形式に変更。アカウント名から直接 URL を構築できるため、実行時の動的取得が不要になる。

🔄

新規作成 + データ移行が必要

既存アカウントのエンドポイントタイプを直接変更することはできない。Standard エンドポイントで新規ストレージアカウントを作成し、AzCopy 等でデータを移行する必要がある。

🛡

アプリ・ネットワーク構成の更新

アプリケーションコード、ファイアウォールルール、DNS 設定、SAS トークンなど、AzureDnsZone エンドポイント URL を参照している全ての箇所を更新する必要がある。

📊

アカウント数の上限変更

Standard エンドポイントでは 1 リージョン・1 サブスクリプションあたり既定 250、クォータ増加申請で最大 500 アカウントに制限される。500 以上必要な場合は複数サブスクリプションの分散やサポートへの相談が必要。

📅

移行猶予期間

2026 年 4 月のアナウンスから 2027 年 3 月の廃止まで約 11 か月の猶予期間がある。計画的に移行を進めることが推奨される。

⏰ 廃止スケジュール

2026 年 4 月 2 日
廃止アナウンス公開。移行計画の策定を開始すべき時期。
2027 年 3 月
AzureDnsZone エンドポイント (プレビュー) 廃止。この日までにすべてのアカウントを Standard エンドポイントへ移行する必要がある。

✅ 必要なアクション

  1. 影響範囲の特定 - AzureDnsZone エンドポイントを使用しているストレージアカウントを特定する
  2. Standard エンドポイントへの移行 - 新規ストレージアカウントを Standard エンドポイントで作成し、データを移行する
  3. アプリケーションコードの更新 - エンドポイント URL の参照箇所、ファイアウォールルール、DNS 設定、SAS トークンを更新する

💻 対応方法 (コードサンプル)

AzureDnsZone エンドポイント使用状況の確認

Bash
az storage account list --query "[].{name:name, primaryEndpoints:primaryEndpoints.blob}" -o table
az storage account show --name <StorageAccountName> --resource-group <ResourceGroupName> --query "primaryEndpoints"

新規ストレージアカウントの作成 (Standard エンドポイント)

Bash
az storage account create \
  --name <StorageAccountName> \
  --resource-group <ResourceGroupName> \
  --location <Region> \
  --sku Standard_LRS \
  --kind StorageV2

データ移行 (AzCopy)

Bash
azcopy copy "https://<source>.z[XX].blob.storage.azure.net/<container>" \
  "https://<destination>.blob.core.windows.net/<container>" \
  --recursive

💡 ユースケース

1. 少数のストレージアカウントを使用しているケース

プレビュー機能の検証目的で数個の AzureDnsZone アカウントを使用している場合。

→ Standard エンドポイントで新規作成 + AzCopy でデータ移行

比較的少量のデータ移行で済むため、短期間での移行が可能。アプリケーション側のエンドポイント URL 更新も限定的。

2. 大量のストレージアカウントを運用しているケース

マルチテナントアプリケーションで 500 以上のアカウントを 1 リージョンで運用している場合。

→ クォータ確認 + 複数サブスクリプション分散 + Azure サポート相談

Standard エンドポイントの上限 (最大 500) を超える場合は、複数サブスクリプションへの分散やアーキテクチャの見直しが必要。Azure サポートへの早期相談を推奨。

⚠ デメリット・制約事項

  • Standard エンドポイントではアカウント数上限が 250 (最大 500) に制限される。5,000 アカウントが必要な場合は代替策の検討が必要。
  • 既存のアカウントのエンドポイントタイプを直接変更することはできない。新規作成 + データ移行が必要。
  • エンドポイント URL 変更に伴い、アプリケーション、ファイアウォール、DNS、SAS トークンの全てを更新する必要がある。
