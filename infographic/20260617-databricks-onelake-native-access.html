OneLake ネイティブアクセス
読み取り GA / 書き込みプレビュー

Unity Catalog を通じた OneLake データへの双方向ネイティブアクセスが大幅拡充

読み取り: GA 書き込み: Preview Analytics 2026-06-17

概要

Azure Databricks において、Microsoft OneLake に対するネイティブアクセス機能が大幅に拡充された。Unity Catalog を通じた OneLake データへのネイティブ読み取りアクセスが一般提供 (GA) となり、さらに Azure Databricks のマネージド Delta テーブルを OneLake にネイティブに書き込む機能がパブリックプレビューとして提供開始された。

読み取り機能 (GA) により、顧客は OneLake に格納されたデータをコピーや移動なしに直接クエリ・分析できるようになり、Fabric で管理されたデータセットへの高速アクセスが実現される。書き込み機能 (プレビュー) により、OneLake を Azure Databricks ワークロードの統合ストレージレイヤーとして使用でき、別途ストレージアカウントを管理する必要がなくなる。

Key Points

  • OneLake 読み取りアクセスが GA (本番環境 SLA 保証)
  • Databricks マネージド Delta テーブルの OneLake ネイティブ書き込みがプレビュー開始
  • データコピー不要 - Foreign Catalog 経由でスキーマ・テーブルを自動同期
  • OneLake を統合ストレージレイヤーとして利用可能 (ADLS アカウント個別管理不要)
  • Fabric ワークロードから Databricks データに直接アクセス可能

Before (課題)

  • OneLake Catalog Federation では読み取り専用アクセスに限定
  • Databricks のマネージドテーブルを Fabric 側から利用するにはデータコピーが必要
  • Databricks と Fabric で別々のストレージアカウントを管理する必要あり
  • OneLake の読み取りアクセスはプレビュー段階

After (改善)

  • OneLake 読み取りアクセスが GA (本番環境 SLA 保証)
  • Databricks から OneLake へのネイティブ書き込み可能 (双方向データ統合)
  • OneLake を統合ストレージレイヤーとして利用 (ストレージ個別管理不要)
  • Fabric ワークロードから Databricks データに直接アクセス可能

主要機能

📚

OneLake ネイティブ読み取りアクセス

Unity Catalog を通じて OneLake データを直接クエリ・分析。データコピー不要、Foreign Catalog 経由でスキーマ・テーブルを自動同期。GA で本番ワークロード正式サポート。

GA
✍

OneLake ネイティブ書き込み

Managed Delta テーブルを OneLake にネイティブ格納。Azure Data Lake Storage アカウント不要。Fabric ワークロードから直接利用可能。

Preview
🔗

統合ストレージレイヤーとしての OneLake

単一ストレージ基盤で一元管理。データサイロ解消・運用簡素化。Unity Catalog ガバナンス適用により、セキュリティとコンプライアンスを維持。

コード例

SQL Reading (GA) - OneLake データの読み取り 
-- OneLake への接続を作成
CREATE CONNECTION onelake_conn TYPE onelake
OPTIONS (
  workspace '<fabric-workspace-id>',
  credential '<storage-credential-name>'
);

-- Foreign Catalog を作成して Fabric データにアクセス
CREATE FOREIGN CATALOG fabric_data USING CONNECTION onelake_conn
OPTIONS (
  data_item '<fabric-lakehouse-id>',
  item_type 'Lakehouse'
);

-- OneLake のデータを直接クエリ
SELECT * FROM fabric_data.<schema>.<table>;
SQL Writing (Preview) - OneLake へのネイティブ書き込み 
-- OneLake をマネージドロケーションとしてカタログを作成
CREATE CATALOG databricks_onelake
MANAGED LOCATION 'abfss://<workspace-id>@onelake.dfs.fabric.microsoft.com/<lakehouse-id>.lakehouse/Tables';

-- OneLake にテーブルを作成 (書き込み)
CREATE TABLE databricks_onelake.default.my_table AS
SELECT * FROM source_data;

ユースケース

1. Fabric BI + Databricks ML 統合データ基盤

Databricks で ML 推論結果を OneLake に直接書き込み、Power BI ダッシュボードに即時反映。データの移動やコピーが不要なため、リアルタイム性の高い分析基盤を構築できる。

効果: ML 推論結果の可視化までのリードタイム大幅短縮

2. ストレージ統合によるデータサイロ解消

OneLake を標準化ストレージとして採用し、Databricks と Fabric の両方から統一的にアクセス。管理対象のストレージアカウント削減とガバナンス統合を実現。

効果: 管理対象ストレージ削減・運用コスト低減

3. 段階的な Fabric + Databricks 統合

まず読み取り (GA) で即座に利用開始し、Fabric 管理データへのアクセスを確保。次のステップとして書き込み (Preview) を段階的に導入し、完全な双方向統合を達成。

効果: リスクを最小化しながら段階的にデータ統合を推進

制約事項

  • 書き込みはプレビュー (SLA 確認必要)
  • 複合データ型 (配列、マップ、構造体) 非サポート
  • マテリアライズドビュー・ビュー読み取り対象外
  • Runtime 18.0+ 必須
  • バンドルモデルの場合コストが ADLS より高い可能性
  • ワンライターパターン推奨
