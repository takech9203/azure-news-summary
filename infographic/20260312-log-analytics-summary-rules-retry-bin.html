Log Analytics サマリールール
手動 "Retry bin" が一般提供開始

サマリールールで失敗した特定の bin を手動で再実行できる "Retry bin" 機能が GA。API / PowerShell / ARM テンプレートで操作可能。データギャップを修復し、集約データの完全性を回復。

📋 概要
サマリールールは、Log Analytics ワークスペース内でバッチ集計を実行し、集約結果をカスタム宛先テーブルに再取り込みする機能。bin が失敗した場合にデータギャップが発生する問題を、"Retry bin" 機能で手動修復できるようになった。

Before

  • 自動リトライ (最大 10 回) 失敗後、bin レベルの障害修復手段が限定的
  • 集約データにギャップが生じても特定 bin の手動再実行が困難
  • 8 回連続 bin リトライ失敗でルールが停止 (isActive: false)

After

  • API / PowerShell / ARM テンプレートで特定 bin を指定して手動再実行可能
  • retryBinStartTime パラメータで再実行する bin 開始時刻を正確に指定
  • LASummaryLogs テーブルで失敗 bin を特定しピンポイント修復
  • データの完全性を回復しサマリーレポートの信頼性を維持

運用担当者は LASummaryLogs で失敗 bin を確認し、API 経由で Retry bin を実行してデータギャップを修復する

⭐ 主要機能
🔄

手動 Retry bin

失敗した bin を retryBinStartTime パラメータで指定して手動再実行。REST API (PUT) / PowerShell / ARM テンプレートの 3 通りで実行可能。binStartTime は binSize の区切りに合致する必要あり。

🔍

失敗 bin の監視と検出

診断設定で Summary Logs カテゴリを有効化すると、LASummaryLogs テーブルに実行詳細が記録される。Status == "Failed" クエリで失敗 bin を検出。ログアラートルールによるプロアクティブ通知も可能。

📈

データ完全性の検証

make-series と dcount を使用したクエリで特定期間内の bin 実行状況を可視化。timechart で表示しデータギャップを視覚的に確認可能。

🛠 Retry bin ワークフロー
1

診断設定で Summary Logs を有効化

Log Analytics ワークスペースの診断設定で Summary Logs カテゴリを有効にして、LASummaryLogs テーブルへのログ記録を開始する。

2

失敗した bin を特定

LASummaryLogs | where Status == "Failed" クエリで失敗 bin の BinStartTime を取得する。

3

retryBinStartTime で再実行

REST API / PowerShell / ARM テンプレートで retryBinStartTime パラメータに失敗 bin の開始時刻を指定して再実行する。

4

データ完全性を確認

make-series クエリで timechart を表示し、データギャップが修復されたことを確認する。

💻 設定方法・コード例

失敗 bin の検出 (KQL)

KQL 
LASummaryLogs
| where RuleName == "SigninLogsSummary"
| where Status == "Failed"
| project BinStartTime, Status, FailureReason

REST API で Retry bin を実行

bash 
curl -X PUT \
  "https://management.azure.com/subscriptions/{subId}/resourceGroups/{rg}/providers/Microsoft.OperationalInsights/workspaces/{ws}/summarylogs/{ruleName}?api-version=2025-07-01" \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{"properties": {"retryBinStartTime": "2026-03-01T02:00:00Z"}}'

データ完全性の確認 (KQL)

KQL 
let ruleName = "ContainerLogsSummary";
let stepSize = 60m;
LASummaryLogs
| where RuleName == ruleName
| where Status == 'Succeeded'
| make-series dcount(BinStartTime) default=0
    on BinStartTime from datetime("2026-03-01") to datetime("2026-03-12") step stepSize
| render timechart

💡 ユースケース

🛡 セキュリティ監査レポートのデータギャップ修復

セキュリティチームが月次監査レポート用に SigninLogs のサマリーを集計しているが、一時的なサービス障害により特定の時間帯の bin が失敗し、レポートにギャップが発生した。LASummaryLogs で失敗 bin を特定し、Retry bin で再実行する。

効果: 監査レポートのデータ完全性を回復し、コンプライアンス要件を満たすことができる。

💵 コスト最適化のためのサマリーデータ修復

大量のコンテナログを Basic テーブルに保存し、サマリールールで集約データを Analytics テーブルに送信している。bin 失敗によりコスト分析に使う集約データが欠損した場合に、Retry bin で修復する。

効果: 集約データのギャップを埋めることで、正確なコスト分析と最適化の意思決定が可能になる。

⚠ 制限事項・注意点
⚠ retryBinStartTime は RuleLastModifiedTime 以降の時刻でなければならず、ルール変更前の古い bin は再実行不可
⚠ binStartTime は binSize の区切りに正確に一致する必要がある
⚠ サマリールールはパブリッククラウドでのみ利用可能 (Azure Government / Azure China は対象外)
⚠ Lighthouse を介したクロステナントクエリのサマリールールには非対応
⚠ 宛先テーブルへのワークスペース変換 (workspace transformation) の追加は非サポート
