Azure Web Application Firewall (WAF)

Exceptions 機能のパブリックプレビュー

Application Gateway / Front Door 対応 -- より精密な条件ベースの誤検知制御を実現

In Preview 2026-07-07

概要

Azure Web Application Firewall (WAF) に新しい「Exceptions (例外)」機能がパブリックプレビューとして追加された。従来の「Exclusions (除外)」を発展させ、より精密で条件に基づいた誤検知の制御を可能にする。セキュリティ保護レベルを維持しながらアプリケーションの可用性を向上させる機能であり、Azure Application Gateway と Azure Front Door の両方で統一的に利用可能。

Key Points

  • 従来の Exclusions よりも精密な条件ベースの誤検知制御が可能
  • Exclusions を置き換えるのではなく補完する新機能
  • Azure Application Gateway と Azure Front Door の両方で統一的に利用可能
  • セキュリティ保護品質を維持しながら正当なリクエストを柔軟に許可

Before (課題)

  • Exclusions はリクエスト属性 (ヘッダー、Cookie、クエリ文字列、ボディ) 単位の除外のみ
  • 適用範囲の細かい制御に限界があった
  • 誤検知回避のために広範な除外を設定するとセキュリティの隙間が生じるリスク
  • 条件付きの例外処理には非対応

After (改善)

  • Exceptions により既存の Exclusions よりも精密な誤検知制御が可能
  • Application Gateway と Front Door の両方で統一的に利用可能
  • セキュリティ保護品質を維持しながら正当なリクエストを柔軟に許可
  • 条件ベースの例外処理で必要最小限のスコープに限定

WAF エンジン内でマネージドルール評価後、Exclusions と Exceptions が順に適用される。Exceptions は条件ベースでより精密な例外処理を提供。

Key Features

🛡

Exceptions (例外) 機能 NEW

既存 Exclusions とは異なる新しい誤検知軽減メカニズム。属性ベースではなく、より精密な条件ベースの例外処理を実現。誤検知を最小限のスコープで解消しながら、WAF のセキュリティ保護レベルを維持する。

🔗

既存 Exclusions との共存

Exclusions を置き換えるのではなく補完する機能として設計。既存の Exclusions 設定はそのまま継続利用可能で、必要に応じて Exceptions を追加することで段階的にセキュリティ精度を向上できる。

🌐

Application Gateway & Front Door 両対応

Azure Application Gateway と Azure Front Door の両方の WAF ポリシーで統一的に利用可能。デプロイ先に関わらず同一の Exceptions 機能を活用でき、マルチアーキテクチャ環境でも一貫した誤検知対策が可能。

Exclusions vs Exceptions 比較

Exclusions (既存) Exceptions (NEW)
方式 属性ベースの除外 条件ベースの例外
対象 ヘッダー / Cookie / クエリ / ボディ より精密な条件指定
精度 広範囲 ピンポイント
リスク 除外範囲が広いとセキュリティ低下 セキュリティ維持しつつ誤検知軽減

Use Cases

UC1: API リクエストの誤検知対応

REST API のリクエストボディに含まれる SQL ライクな文字列 (例: フィルター条件やクエリパラメータ) が WAF の SQL インジェクションルールにマッチし、正当なリクエストがブロックされるケース。

-- Exceptions を使用して、特定の API エンドポイントと特定のルール ID に対してのみ精密な例外を定義。広範な Exclusion を設定する必要がなくなり、他のエンドポイントのセキュリティは維持される。

UC2: 認証トークンの誤検知対応

Azure AD 認証トークンに含まれる特殊文字 (Base64 エンコードされた JWT など) が WAF のクロスサイトスクリプティングやインジェクション検出ルールにマッチし、認証フローが中断されるケース。

-- Exceptions により認証ヘッダーに対する条件付き例外を定義。トークン固有のパターンに限定した例外処理で、認証フローを維持しつつ他のヘッダーに対するセキュリティ検査は継続。
