Azure Storage GPv1 / Legacy Blob アカウント廃止

GPv2 へのインプレースアップグレードでダウンタイムなし移行

📅 発表日: 2026-06-11 ⚠️ 廃止期限: 2026-10-13 📄 カテゴリ: Storage

📋 概要

Microsoft は Azure Storage の GPv1 (General Purpose v1) アカウントおよびレガシー Blob ストレージアカウントの完全廃止を 2026 年 10 月 13 日に予定しています。対象アカウントを使用中のお客様は、GPv2 (General Purpose v2) へのインプレースアップグレードを期限までに実施する必要があります。
⚠ 重要: 期限までに移行が完了しない場合、Microsoft が自動的にアカウントを GPv2 にアップグレードします。予期しない動作やコスト変動を避けるため、計画的な移行を推奨します。

⚠ Before (廃止対象)

  • GPv1 ストレージアカウント (kind: Storage)
  • レガシー Blob ストレージ (kind: BlobStorage)
  • Blob 階層化なし (Hot のみ)
  • ライフサイクル管理なし
  • ZRS/GZRS 非対応
  • 不変ストレージ非対応

✔ After (移行先: GPv2)

  • GPv2 ストレージアカウント (kind: StorageV2)
  • Blob 階層化 (Hot/Cool/Cold/Archive)
  • ライフサイクル管理ポリシー
  • ZRS/GZRS/RA-GZRS 冗長性
  • 不変ストレージ (WORM)
  • 最新の全機能にアクセス可能

GPv1 / Legacy Blob から GPv2 への移行フローと GPv2 で利用可能になる主要機能

⭐ GPv2 の主要機能

🌡

Blob 階層化 (Hot/Cool/Cold/Archive)

データのアクセス頻度に応じて最適なストレージ階層を選択し、コストを最適化できます。アクセス頻度の低いデータを Cool/Cold/Archive に移動することで、最大 80% のストレージコスト削減が可能です。

♻

ライフサイクル管理

ポリシーベースのルールにより、Blob を自動的に適切な階層に移動または削除できます。データ保持ポリシーの自動化により運用コストを削減します。

🛡

高度な冗長性オプション (ZRS/GZRS)

ゾーン冗長ストレージ (ZRS) やジオゾーン冗長ストレージ (GZRS) をサポートし、データセンター障害やリージョン障害に対する耐障害性を向上させます。

🔒

不変ストレージ (WORM)

Write Once, Read Many (WORM) ポリシーにより、規制要件やコンプライアンス要件に対応したデータの不変保持が可能です。金融・医療分野で必須の機能です。

⚡

パフォーマンスの向上

GPv2 では最大 Ingress 60 Gbps、Egress 200 Gbps のスループットをサポートし、高負荷ワークロードにも対応します。また、最新の API 機能をすべて利用可能です。

💻 移行コマンド

Azure CLI を使用したインプレースアップグレードの実行例:

Azure CLI 
# 1. 対象アカウントの確認 (Azure Resource Graph)
az graph query -q "
  resources
  | where type == 'microsoft.storage/storageaccounts'
  | where kind in ('Storage', 'BlobStorage')
  | project name, resourceGroup, kind, location
"

# 2. GPv2 へのインプレースアップグレード (ダウンタイムなし)
az storage account update \
  --name <storage-account-name> \
  --resource-group <resource-group> \
  --set kind=StorageV2

# 3. アップグレード後の確認
az storage account show \
  --name <storage-account-name> \
  --resource-group <resource-group> \
  --query "{name:name, kind:kind, sku:sku.name}"
💡 ヒント: PowerShell を使用する場合は Set-AzStorageAccount -ResourceGroupName <rg> -Name <name> -UpgradeToStorageV2 コマンドを使用します。Azure Portal からも「構成」ブレードでアップグレードが可能です。

💡 移行ユースケース

1

コスト最適化 - データ階層化による削減

アクセス頻度の低い大量のログデータやバックアップデータを保持している環境。GPv2 移行後にライフサイクルポリシーを適用し、30 日後に Cool、90 日後に Cold、365 日後に Archive へ自動移行。

✔ ストレージコストを最大 60-80% 削減しながら、必要時にはデータへのアクセスを維持
2

ディザスタリカバリ強化 - ZRS/GZRS 導入

金融系システムのデータを LRS で保持しているが、データセンター障害時の可用性要件を満たせていない。GPv2 移行後に ZRS または GZRS に変更し、複数の可用性ゾーンにデータを分散。

✔ 可用性 SLA が 99.9% から 99.99% に向上し、ゾーン障害に対する耐障害性を確保
3

コンプライアンス対応 - 不変ストレージ

医療データや金融取引記録の保持に規制要件があり、データの改ざん防止が必要。GPv2 の不変ストレージ (WORM) ポリシーを適用し、指定期間中のデータ削除・変更を防止。

✔ SEC 17a-4、CFTC、FINRA 等の規制要件に準拠したデータ保持を実現
