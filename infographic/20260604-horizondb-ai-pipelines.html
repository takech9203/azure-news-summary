AI Pipelines

SQL で宣言的に定義する AI データ取り込みワークフロー

Public Preview 2026-06-04

📋 概要

AI Pipelines は、AI データ取り込みワークフロー（チャンキング、埋め込み生成、抽出、生成、ランキング）を SQL で宣言的に定義し、データベースエンジン内部で pg_durable によるフォールトトレラントなパイプラインとして実行する新機能です。外部オーケストレーター（Azure Functions、Airflow）が不要になります。
宣言的定義: SQL でパイプラインを定義、外部オーケストレーター不要
耐久実行: pg_durable によるクラッシュ復旧、リトライ、チェックポイント
5 種類の AI ステップ: chunk, embed, extract, generate, rank
差分処理: incremental_column で変更行のみを効率的に処理
モデル変更対応: ai.backfill() で安全な一括再処理
トリガーモード: on_change（自動）と manual（手動）
5 AI ステップタイプ
2 トリガーモード
512 デフォルト Chunk Size

Before / After 比較

❌ Before（従来の課題）

  • API 障害でバッチ処理中断時、共有チェックポイントがない
  • ワーカークラッシュでインデックス不整合の可能性
  • モデル変更時に変更行だけを再 Embed する手段がない
  • 外部オーケストレーターの追加運用コスト

✅ After（AI Pipelines）

  • 定義・実行状態・結果すべて SQL で管理
  • pg_durable による自動クラッシュ復旧・リトライ
  • ai.backfill() でチェックポイント付き再処理
  • incremental_column で効率的な差分処理

⭐ 主要機能

📝 宣言的パイプライン定義

ai.create_pipeline() で SQL によりパイプラインを定義。ソース、ステップ（配列）、シンク、トリガーを宣言的に指定し、システムカタログに行として保存されます。

🧩 5 種類の AI ステップ

  • ai.chunk() - テキストをオーバーラップ付きチャンクに分割
  • ai.embed() - ベクトル埋め込みを生成
  • ai.extract() - LLM を使用して構造化フィールドを抽出
  • ai.generate() - プロンプトテンプレートからテキスト生成
  • ai.rank() - ドキュメントをクエリに対してスコアリング

🛡 耐久実行（pg_durable ベース）

クラッシュ復旧（最後のチェックポイントから再開）、失敗ステップの自動リトライ、インクリメンタルなチェックポイント管理を提供。データベース再起動後も安全に処理を継続します。

⚡ トリガーモード

on_change: ソーステーブルの変更を検知して自動実行。manual: ai.run() の明示的呼び出しで実行。用途に応じた柔軟な実行制御が可能です。

🔄 モデル変更対応（ai.backfill）

ai.backfill() で全行の再処理を安全に実行。チェックポイント付きで実行されるため、大量データの再処理中に中断しても、最後の完了位置から再開可能です。

💻 コードサンプル

RAG パイプラインの定義例 - チャンキングと Embedding 生成を自動化

-- 必要な拡張機能を有効化
CREATE EXTENSION IF NOT EXISTS pg_durable;
CREATE EXTENSION IF NOT EXISTS azure_ai;
CREATE EXTENSION IF NOT EXISTS vector;

-- パイプラインを定義
SELECT ai.create_pipeline(
    name   => 'rag_pipeline',
    source => ai.table_source(
        table_name => 'documents'
    ),
    steps  => ARRAY[
        ai.chunk(input      => 'content',
                 chunk_size => 512,
                 overlap    => 64),
        ai.embed(model      => 'default-embedding',
                 input      => 'chunk_text',
                 dimensions => 1536)
    ],
    trigger => 'on_change',
    sink   => ai.table_sink('rag_pipeline_output')
);

-- パイプラインを実行
SELECT ai.run('rag_pipeline');

💡 ユースケース

📚 ユースケース 1: RAG パイプラインの構築

ドキュメント管理システムに新規文書がアップロードされるたびに、自動的にチャンキング・埋め込み生成を行い、ベクトル検索に対応させるシナリオ。

効果: 新規ドキュメント追加時に自動でチャンキングと Embedding 生成が実行され、障害時もチェックポイントから安全に再開。外部サービスの構築が不要。

📄 ユースケース 2: 構造化データ抽出 + Embedding

非構造化テキスト（契約書、レポート）から構造化情報を LLM で抽出し、同時に検索用の Embedding も生成する複合パイプライン。

効果: 抽出と Embedding 生成を単一パイプラインで実行。LLM API の一時障害があっても、完了済みのチャンキングステップは再実行されない。
