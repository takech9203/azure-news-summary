リージョンで一般提供開始

Azure File Sync が新たに 3 リージョンで GA。ヨーロッパおよび東南アジアにおけるハイブリッドファイル共有のカバレッジを強化し、各国のデータ居住要件 (EU GDPR, Malaysia PDPA, Indonesia PDP Law) への対応を実現。

📋 概要
Azure File Sync が Belgium Central、Malaysia West、Indonesia Central の 3 つの新リージョンで一般提供 (GA) を開始。オンプレミスの Windows Server から Azure Files へのハイブリッドファイル共有とクラウド階層化を、より近いリージョンで利用可能に。SMB、NFS、FTPS プロトコルによるローカルアクセスをサポートし、データ居住要件への準拠とハイブリッドファイルサーバーのパフォーマンス向上を両立。
  • Azure File Sync が Belgium Central、Malaysia West、Indonesia Central の 3 リージョンで GA
  • オンプレミス Windows Server と Azure Files 間のハイブリッドファイル共有をクラウド階層化付きで実現
  • SMB、NFS、FTPS プロトコルによるローカルアクセスをサポート
  • ヨーロッパおよび東南アジアにおけるカバレッジを強化
  • 各国のデータ居住要件 (EU GDPR、Malaysia PDPA、Indonesia PDP Law) への対応

Before (従来)

  • ベルギー、マレーシア、インドネシアの顧客は遠隔リージョンに Storage Sync Service をデプロイする必要があり、同期レイテンシが高い
  • データ居住要件・コンプライアンス要件の充足が困難
  • 物理的な距離がハイブリッドファイルサーバーのパフォーマンスに悪影響

After (今後)

  • 3 つの新リージョンに Storage Sync Service をローカルデプロイ可能、同期レイテンシを大幅に削減
  • 各国・リージョンのデータ居住要件に準拠
  • ディザスター リカバリーやマルチサイト同期においてより柔軟なリージョン選択が可能に

⭐ 主要機能
🌍

3 リージョンでの GA 提供

Belgium Central (EU 圏)、Malaysia West (東南アジア)、Indonesia Central (インドネシア初) の 3 リージョンで一般提供開始。各地域のデータ居住要件に対応しつつ、ハイブリッドファイル共有の同期レイテンシを大幅に削減。

☁

Cloud Tiering (クラウド階層化)

頻繁にアクセスされるファイルをローカルにキャッシュし、アクセス頻度の低いファイルを自動的に Azure Files へ階層化。オンプレミスのストレージ使用量を最適化しつつ、すべてのファイルへのシームレスなアクセスを維持。

🔄

マルチサイトアクセスと同期

複数の登録済み Windows Server 間で双方向同期を実現。同期グループ内の各サーバーエンドポイントがクラウドエンドポイント (Azure Files) を介してファイルを共有し、拠点間のコラボレーションを強化。

🛡

事業継続性とディザスター リカバリー

Azure Files の冗長性オプション (LRS、ZRS、GRS、GZRS) を活用したデータ保護。新リージョンの追加により、ディザスター リカバリーのリージョンペアの選択肢が拡大。

📦

クラウド移行

オンプレミスから Azure Files へのシームレスな段階的移行を実現。Cloud Tiering を活用しながら段階的にクラウドへ移行し、最終的にオンプレミスサーバーを廃止可能。

💡 ユースケース

🏢 ヨーロッパのハイブリッドファイルサーバー

Belgium Central リージョンを活用し、EU GDPR に準拠したハイブリッドファイルサーバーを構築。ベルギーおよび EU 圏内の拠点から低レイテンシでファイルにアクセス。

効果: データを EU 圏内に保持しつつ、クラウド階層化でストレージコストを最適化。GDPR 対応の監査要件にも対応可能。

🌐 東南アジアのマルチサイトファイル共有

Malaysia West と Indonesia Central の 2 リージョンを組み合わせ、東南アジア複数拠点間でのファイル共有を実現。各国のデータ居住要件を満たしながらコラボレーションを強化。

効果: マレーシア PDPA およびインドネシア PDP Law に準拠しつつ、拠点間の双方向同期でリアルタイムなファイル共有が可能に。

💾 オンプレミスストレージ最適化

Cloud Tiering を活用し、アクセス頻度の低いファイルを自動的に Azure Files へ階層化。オンプレミスのストレージ容量を削減しながら、ユーザーには透過的なアクセスを維持。

効果: オンプレミスのストレージ投資を削減。ユーザーはファイルの物理的な保存場所を意識せずにすべてのファイルにアクセス可能。

🔀 段階的なクラウド移行

Azure File Sync を使用してオンプレミスのファイルサーバーから Azure Files へ段階的に移行。Cloud Tiering でデータを徐々にクラウドへ移行し、最終的にオンプレミスサーバーを廃止。

効果: 業務中断なく段階的にクラウドへ移行でき、移行完了後はオンプレミスのハードウェア管理コストを完全に削減。

🛡 地理冗長ストレージによる DR

Azure Files の GRS / GZRS 冗長オプションと新リージョンを組み合わせ、地理的に分散されたディザスター リカバリー構成を実現。主要データを近隣リージョンに同期しつつ、遠隔リージョンにレプリカを保持。

効果: リージョン障害時にもデータ損失を最小限に抑え、迅速な復旧が可能。新リージョンの追加により DR の選択肢がさらに柔軟に。

🛠 セットアップ (PowerShell)

新リージョン (例: Belgium Central) に Storage Sync Service と同期グループを作成する手順:

PowerShell 
# 新リージョンに Storage Sync Service を作成
$region = "belgiumcentral"
$resourceGroup = "<リソースグループ名>"
$storageSyncName = "<Storage Sync Service 名>"

New-AzStorageSyncService `
    -ResourceGroupName $resourceGroup `
    -Name $storageSyncName `
    -Location $region

# 同期グループを作成
New-AzStorageSyncGroup `
    -ResourceGroupName $resourceGroup `
    -StorageSyncServiceName $storageSyncName `
    -SyncGroupName "<同期グループ名>"
Malaysia West / Indonesia Central の場合: $region の値を "malaysiawest" や "indonesiacentral" に変更するだけで、同じ手順でデプロイ可能です。

デプロイ手順

1

Storage Sync Service の作成

Azure Portal または PowerShell で新リージョンに Storage Sync Service リソースを作成

2

同期グループの作成

Storage Sync Service 内に同期グループを作成し、Cloud Endpoint (Azure ファイル共有) を登録

3

Azure File Sync Agent のインストール

オンプレミスの Windows Server に Azure File Sync Agent をインストールし、Storage Sync Service に登録

4

Server Endpoint の追加

登録済みサーバーのローカルパスを Server Endpoint として同期グループに追加。Cloud Tiering ポリシーを設定

5

同期の開始

初回同期が自動的に開始。以降はリアルタイムで双方向同期が実行される

⚠ 制限事項・注意点
⚠ NTFS のみ: ReFS、FAT、FAT32 はサポートされない。ファイルシステムは NTFS ボリュームである必要がある
⚠ 最大 30 共有: 1 サーバーあたり最大 30 の Azure ファイル共有に制限
⚠ Cloud Endpoint の直接変更検出: Azure ファイル共有への直接変更の検出は 24 時間に 1 回
⚠ Windows Server のみ: Linux および macOS はサポート対象外
⚠ インターネットルーティング不可: Microsoft ルーティングが必須 (インターネットルーティングは非対応)
⚠ 特殊ファイルはスキップ: ハードリンク、シンボリックリンク、リパースポイントは同期対象外
⚠ データベースファイル非推奨: データベースファイルの同期は推奨されない (ファイルロックの問題が発生する可能性)
