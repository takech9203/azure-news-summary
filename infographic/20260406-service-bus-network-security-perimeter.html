Azure Service Bus GA (一般提供) Security Integration

Network Security Perimeter の一般提供開始

リリース日: 2026-04-06 機能: Network Security Perimeter (NSP) サポート

概要

Azure Service Bus における Network Security Perimeter (NSP) のサポートが一般提供 (GA) となった。Network Security Perimeter は、Azure PaaS リソースの周囲に論理的なネットワーク境界を作成し、不正なパブリックアクセスをブロックする機能である。

Service Bus 名前空間を NSP に関連付けることで、同一ペリメーター内の PaaS リソース間のみに通信が制限される。外部との通信が必要な場合は、明示的なアクセスルールを設定することで許可できる。ペリメーター内通信はデフォルトで許可、外部アクセスはデフォルトで拒否される。

Before (従来)

  • IP ファイアウォール、VNet エンドポイント、プライベートエンドポイントなど個別機能に依存
  • 複数 PaaS 間の一元的セキュリティ管理なし
  • Data Exfiltration リスクの統合管理困難

After (NSP 導入後)

  • 論理的なネットワーク境界で複数 PaaS を統合保護
  • ペリメーター内通信はデフォルト許可、外部はデフォルト拒否
  • アクセスログによる監査・コンプライアンス対応
  • Portal / PowerShell / CLI での一元管理

Network Security Perimeter は Service Bus 名前空間を含む PaaS リソースの周囲に論理境界を形成する。ペリメーター内のリソース間通信はデフォルトで許可され、外部からのパブリックアクセスはアクセスルールによって制御される。プライベートエンドポイント経由のトラフィックは NSP ルールの対象外として常に許可される。

主要機能

1. ペリメーター内リソース間通信

同一 NSP 内に関連付けられた PaaS リソース間の通信がデフォルトで許可される。Service Bus と Azure Key Vault 間の CMK 通信など、PaaS 間連携をセキュアに実現。

2. パブリックアクセスの明示的制御

Enforced モードではペリメーター外からのパブリックトラフィックがデフォルトで拒否される。インバウンドルール (IP ベース / サブスクリプションベース) およびアウトバウンドルール (FQDN ベース) で例外を設定可能。

3. アクセスモード (Transition / Enforced)

Transition モード (旧 Learning モード) で既存のアクセスパターンを分析し、準備が整ったら Enforced モードに移行。段階的な導入が可能。

4. 診断ログ (NSPAccessLogs)

NSPAccessLogs テーブルを通じてアクセスログを収集可能。インバウンド/アウトバウンドの許可・拒否の各カテゴリでログを取得。Log Analytics、Azure Storage、Event Hubs へのログ送信に対応。

5. Service Bus 固有シナリオ対応

カスタマーマネージドキー (CMK) のための Azure Key Vault との通信をサポート。Key Vault を同一ペリメーター内に配置することで、暗号化キーへのセキュアなアクセスを実現。

設定方法 (Azure CLI)

Network Security Perimeter の作成と関連付け

# Network Security Perimeter の作成
az network perimeter create \
  --name myNSP \
  --resource-group myResourceGroup \
  --location japaneast

# プロファイルの作成
az network perimeter profile create \
  --perimeter-name myNSP \
  --resource-group myResourceGroup \
  --name myProfile

# Service Bus 名前空間をペリメーターに関連付け
az network perimeter association create \
  --perimeter-name myNSP \
  --resource-group myResourceGroup \
  --name myAssociation \
  --access-mode Transition \
  --profile myProfile \
  --private-link-resource "/subscriptions/{subscription-id}/resourceGroups/{rg}/providers/Microsoft.ServiceBus/namespaces/{namespace-name}"

ユースケース

ユースケース 1: メッセージングシステムのデータ漏洩防止

金融機関で Service Bus を使用したメッセージングシステムを運用しており、機密データの外部漏洩を防止する必要がある。

  • Service Bus + Key Vault (暗号化キー管理) + Storage (メッセージアーカイブ) をペリメーターで保護
  • ペリメーター内の PaaS リソース間通信はデフォルトで許可
  • 外部への不正なデータ送信がデフォルトでブロック
  • 診断ログにより全アクセス試行を監査可能
Service Bus と Key Vault がペリメーター内で安全に通信でき、データ漏洩リスクを大幅に軽減

ユースケース 2: 段階的なセキュリティ強化

既存の Service Bus 環境に対して、サービスを停止させることなく段階的に NSP を導入したい場合。

  • まず Transition モードで関連付け (既存トラフィックへの影響なし)
  • 診断ログを有効化してアクセスパターンを分析
  • 必要なアクセスルールを作成
  • 準備が整ったら Enforced モードへ移行
Transition モードでアクセスパターンを把握し、サービス中断リスクを最小化して移行可能

制約事項・注意点

制約事項

  • Geo-Replication 非対応: Service Bus の Geo-Replication 機能は現時点で NSP に対応していない
  • プライベートエンドポイントは NSP ルール対象外: プライベートエンドポイント経由のトラフィックは NSP アクセスルールでは制御されない
  • サービスエンドポイント非対応: サービスエンドポイント経由のトラフィックは NSP ではサポートされない。IaaS - PaaS 間通信にはプライベートエンドポイントの使用を推奨

注意事項

  • SAS トークン認証の制限: ペリメーター内通信およびサブスクリプションベースのインバウンドルールでは SAS トークン認証がサポートされない。Microsoft Entra ID など別の認証方法を使用する必要がある
  • リソース名の長さ制限: NSP リソース関連付けの名前は最大 80 文字のため、リソース名は 44 文字以内に制限される
  • ログ送信先の制約: 診断ログの送信先 (Log Analytics、Storage、Event Hubs) は同一ペリメーター内に存在する必要がある
