Application Gateway for Containers:
Inference Gateway (AI ゲートウェイ機能)

2026-06-24

概要

Application Gateway for Containers に新しい「Inference Gateway」機能がパブリックプレビューとして追加されました。Kubernetes Gateway API Inference Extension を統合し、セルフホストされたモデルサーバーに対するモデル認識ルーティングと負荷認識エンドポイント選択を実現します。Body-Based Router (BBR) が OpenAI 互換リクエストボディからモデル名を抽出し、Endpoint Picker (EPP) がキュー深度や KV キャッシュ使用率に基づいてインテリジェントにエンドポイントを選択。WAF 統合により AI 推論トラフィックのセキュリティも確保します。

Before

  • モデル認識ルーティングにカスタムプロキシ/ルーティングレイヤーが必要
  • モデルサーバーの負荷（キュー深度、KV キャッシュ）に基づくインテリジェントルーティングなし
  • AI エンドポイント向けの統合 WAF + トラフィック管理なし
  • モデルのカナリア/Blue-Green デプロイメントサポートが限定的

After

  • Gateway API Inference Extension (InferencePool/InferenceObjective) で Kubernetes ネイティブ管理
  • BBR がマネージドコンポーネントとして OpenAI 互換リクエストからモデル名を抽出
  • EPP が負荷認識エンドポイント選択（最も負荷の低いレプリカにルーティング）
  • WAF 統合で悪意あるリクエストを GPU バックエンド到達前にブロック
  • HTTPRoute の weighted backendRefs でカナリア/Blue-Green モデルデプロイ

主要機能

🔗

Gateway API Inference Extension サポート

InferencePool と InferenceObjective カスタムリソースによる Kubernetes ネイティブな推論ワークロード管理。Gateway API v1.5 および Inference Extension v1alpha1 に対応。

📋

Body-Based Router (BBR)

マネージドコンポーネントとして OpenAI 互換 API リクエスト (/v1/chat/completions, /v1/completions, /v1/models) からモデル名を抽出し、X-Gateway-Model-Name ヘッダーに変換。

🎯

Endpoint Picker (EPP)

キュー深度と KV キャッシュ使用率に基づく負荷認識エンドポイント選択。最も負荷の低いレプリカにリクエストをルーティングし、Time to First Token (TTFT) を改善。

🔀

モデル認識ルーティング

モデル名に基づくルーティング、HTTPRoute の weighted backendRefs によるトラフィックスプリッティングでカナリアデプロイを実現。

⚡

リクエスト優先度とオーバーロード保護

InferenceObjective による優先度制御（Critical / Default）とリクエストシェディングで、過負荷時でも重要なリクエストを保護。GPU 効率を最大化。

🛡

WAF との統合

OWASP プロテクションを AI 推論トラフィックのエッジで適用。悪意あるリクエストを GPU バックエンドに到達する前にブロックし、セキュリティとコスト効率を両立。

コードサンプル

YAML InferencePool + HTTPRoute 定義 
# InferencePool definition
apiVersion: inference.networking.x-k8s.io/v1alpha1
kind: InferencePool
metadata:
  name: my-llm-pool
spec:
  targetPortNumber: 8000
  selector:
    matchLabels:
      app: vllm-server
  endpointPickerConfig:
    extensionRef:
      name: my-epp
    failureMode: FailOpen
---
# HTTPRoute with InferencePool backend
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: inference-route
spec:
  parentRefs:
  - name: my-gateway
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /v1
    backendRefs:
    - group: inference.networking.x-k8s.io
      kind: InferencePool
      name: my-llm-pool

ユースケース

1. マルチモデルサービングとバージョンロールアウト

複数のモデルバージョンを同時にサービングし、weighted backendRefs を使ったカナリアデプロイで段階的にトラフィックを新バージョンに移行。モデル名ベースのルーティングにより、同一エンドポイントで異なるモデルを提供可能。

2. レイテンシクリティカルなトラフィック優先制御

InferenceObjective を使用してリクエストの優先度を設定（例: チャット = Critical、バッチ処理 = Default）。過負荷時にはリクエストシェディングにより低優先度のリクエストを制御し、重要なリアルタイムトラフィックの Time to First Token (TTFT) を保証。

制約事項・注意点

  • パブリックプレビュー段階 - 本番環境 SLA なし
  • EPP はお客様による手動デプロイ/管理が必要（クリティカルパスコンポーネント）
  • セルフホスト推論のみ対応（パブリック/マネージドモデルプロバイダーは対象外）
  • API バージョン/CRD 互換性の管理が必要
  • BBR はモデル名を抽出できない場合、リクエストを拒否
  • GPU ノードプール、デバイスプラグイン、モデルダウンロード資格情報が必要
