カスケード読み取りレプリカの一般提供開始

2 階層カスケードレプリケーションで最大 30 読み取りレプリカ、プライマリの負荷を軽減しグローバル読み取り分散を実現

Launched (GA) 2026-04-24

概要

Azure Database for PostgreSQL Flexible Server において、カスケード読み取りレプリカ (Cascading Read Replicas) が一般提供 (GA) となった。従来はプライマリサーバーから直接最大 5 つの読み取りレプリカを作成可能であったが、本機能により 2 階層のカスケード構成が可能となり、第 1 レベルのレプリカをソースとして第 2 レベルのカスケードレプリカを作成できる。PostgreSQL ネイティブの非同期物理レプリケーション (ストリーミング + ログシッピング) を使用し、プライマリサーバーへのレプリケーション負荷を大幅に軽減しながら、最大 30 の読み取りレプリカによるスケーラブルな読み取り分散を実現する。

Key Points

  • 2 階層カスケード構成: プライマリ → 第 1 レベル → 第 2 レベル
  • 最大 30 読み取りレプリカ (第 1 レベル 5 + 第 2 レベル 25)
  • PostgreSQL ネイティブ非同期物理レプリケーション使用
  • 中間レプリカとカスケードレプリカ間のスイッチオーバーをサポート
  • クロスリージョンカスケードレプリカ対応
  • PostgreSQL 14 以降、General Purpose / Memory Optimized ティアのみ

Before

  • プライマリから直接最大 5 読み取りレプリカのみ
  • 全レプリカがプライマリに接続 → レプリケーション負荷が集中
  • 地理的に分散した読み取りの柔軟性が限定的

After

  • 2 階層カスケードで最大 30 読み取りレプリカ
  • 第 1 レベルレプリカがソースとなりプライマリの負荷を軽減
  • クロスリージョンカスケードレプリカでグローバル読み取り分散
  • ソースとカスケードレプリカ間のスイッチオーバーをサポート

🔄

2 階層カスケードレプリケーション

プライマリ → 第 1 レベルレプリカ → 第 2 レベルカスケードレプリカの 2 階層構成により、プライマリサーバーへのレプリケーション負荷を分散。第 1 レベルレプリカがレプリケーションソースとなり、プライマリの書き込みパフォーマンスを維持しながら大規模な読み取りスケーリングを実現。

📈

大規模スケーラビリティ (最大 30 レプリカ)

プライマリあたり最大 5 つの第 1 レベルレプリカ、各第 1 レベルレプリカから最大 5 つの第 2 レベルカスケードレプリカを作成可能。合計最大 30 の読み取りレプリカにより、大規模な読み取りワークロードに対応。

🔀

スイッチオーバーサポート

中間レプリカ (第 1 レベル) とカスケードレプリカ (第 2 レベル) の間でスイッチオーバーが可能。ただし、カスケードレプリカを持つ中間レプリカをプライマリに昇格させることはできない点に注意。

🌍

クロスリージョンサポート

カスケードレプリカをプライマリとは異なるリージョンに配置可能。グローバルに分散した読み取りアーキテクチャを構築し、エンドユーザーに近いリージョンから低レイテンシの読み取りを提供。

Azure CLI - カスケードレプリカ作成

# 第 1 レベル読み取りレプリカの作成
az postgres flexible-server replica create \
  --replica-name <replica-name> \
  --resource-group <resource-group> \
  --source-server <primary-server-name>

# 第 2 レベル カスケードレプリカの作成
az postgres flexible-server replica create \
  --replica-name <cascade-replica-name> \
  --resource-group <resource-group> \
  --source-server <level1-replica-name>

Use Cases

UC1: 大規模 BI/分析ワークロードの負荷分散

大量のレポート生成や分析クエリを実行する BI ワークロードにおいて、プライマリサーバーのパフォーマンスに影響を与えずに多数の読み取りクエリを処理するケース。

Implementation:
  • 第 1 レベルに 3-5 台のレプリカを配置し、チーム別に読み取りを分散
  • 第 2 レベルに各レプリカから追加のカスケードレプリカを作成し、重い分析クエリ用に専用リソースを確保
  • プライマリは OLTP トランザクション処理に集中

-- プライマリのレプリケーション負荷を軽減しつつ、最大 30 台のレプリカで読み取りワークロードを分散。

UC2: グローバル分散読み取りアーキテクチャ

複数のリージョンにまたがるグローバルアプリケーションで、各リージョンのユーザーに低レイテンシの読み取りアクセスを提供するケース。

Implementation:
  • 主要リージョンに第 1 レベルレプリカを配置 (例: 東日本、米国東部、西ヨーロッパ)
  • 各リージョン内の第 1 レベルレプリカから、近隣リージョンに第 2 レベルカスケードレプリカを展開
  • エンドユーザーに最も近いレプリカにルーティング

-- クロスリージョンカスケードにより、グローバルな読み取りレイテンシを大幅に削減。

UC3: 段階的ディザスタリカバリ

プライマリリージョンの障害時に、カスケード構成を活用して段階的なフェイルオーバーを行い、業務継続性を確保するケース。

Implementation:
  • DR リージョンに第 1 レベルレプリカを配置
  • DR リージョン内に第 2 レベルカスケードレプリカを追加配置
  • スイッチオーバー機能でカスケードレプリカ間のロール変更を実施

-- 多層構成により、単一リージョン障害時にも複数のフェイルオーバー候補を確保。

Pricing

通常の Flexible Server インスタンスと同じ料金体系

カスケードレプリカは通常の Flexible Server インスタンスとして課金される。コンピューティング (vCores) + ストレージ (GB/月) の従量課金。クロスリージョンレプリカの場合は、リージョン間のデータ転送コストが追加で発生する。

コンピューティングvCores 単位で課金 (General Purpose / Memory Optimized)
ストレージGB/月 単位で課金
クロスリージョンデータ転送リージョン間のデータ転送量に応じた課金

Limitations / Caveats

  • 第 2 レベルのレプリケーションラグが増大 (2 ホップ経由のため)
  • カスケードレプリカを持つ中間レプリカはプライマリに昇格不可
  • カスケードレプリカに対する仮想エンドポイント (Virtual Endpoints) は非対応
  • Burstable ティアは非対応 -- General Purpose / Memory Optimized のみ
  • PostgreSQL 14 以降が必要 -- バージョン 13 以前では利用不可
  • レプリカ上でのバックアップおよび HA は非対応
  • レプリケーションスロットの遅延による WAL ファイル蓄積リスク
  • メジャーバージョンアップグレード前にすべてのカスケード + 読み取りレプリカの削除が必要
