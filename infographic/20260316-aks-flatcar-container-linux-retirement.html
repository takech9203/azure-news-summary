🔄 概要: 何が変わるのか

Flatcar Container Linux は、コンテナワークロードに特化した軽量 Linux ディストリビューションとして AKS のプレビュー機能で提供されていましたが、サポート終了が決定されました。

❌ Before (現状)

  • Flatcar Container Linux (Preview) をノード OS として使用
  • プレビュー機能のため本番ワークロードでは非推奨
  • リタイア後はセキュリティパッチ停止
  • バグ修正の提供も終了

✅ After (移行後)

  • Azure Linux / Ubuntu (GA) でサポート対象 OS に移行
  • 継続的なセキュリティ更新を受けられる
  • Microsoft の公式サポートを利用可能
  • Azure Linux はコンテナ最適化で軽量

🚀 移行手順
1

現在の OS SKU を確認

Azure CLI で対象ノードプールの OS SKU を確認します

2

新しいノードプールを作成

Azure Linux または Ubuntu を OS SKU とするノードプールを追加します

3

ワークロードを移動

nodeSelector や affinity を更新し、ワークロードを新しいノードプールへ移動します

4

古いノードプールを削除

Flatcar Container Linux のノードプールを削除してクリーンアップします

OS SKU の確認コマンド

bash 
# ノードプールの OS SKU を確認する
az aks nodepool show \
    --resource-group <RESOURCE_GROUP> \
    --cluster-name <CLUSTER_NAME> \
    --name <NODEPOOL_NAME> \
    --query osSku

Azure Linux への移行 (推奨)

bash 
# Azure Linux を使用した新しいノードプールを作成する
az aks nodepool add \
    --resource-group <RESOURCE_GROUP> \
    --cluster-name <CLUSTER_NAME> \
    --name <NEW_NODEPOOL_NAME> \
    --os-sku AzureLinux \
    --node-count <NODE_COUNT>

# ワークロードを新しいノードプールに移動する
# (nodeSelector や affinity の更新が必要な場合がある)

# 古い Flatcar ノードプールを削除する
az aks nodepool delete \
    --resource-group <RESOURCE_GROUP> \
    --cluster-name <CLUSTER_NAME> \
    --name <OLD_NODEPOOL_NAME>

Ubuntu への移行

bash 
# Ubuntu を使用した新しいノードプールを作成する
az aks nodepool add \
    --resource-group <RESOURCE_GROUP> \
    --cluster-name <CLUSTER_NAME> \
    --name <NEW_NODEPOOL_NAME> \
    --os-sku Ubuntu \
    --node-count <NODE_COUNT>
Azure Portal からの移行: AKS クラスターページ → 「ノードプール」セクション → 「ノードプールの追加」→ OS SKU として「Azure Linux」または「Ubuntu」を選択 → ワークロード移動 → 古いノードプール削除
💡 ユースケース: 移行先の選び方

🐧 Azure Linux を選ぶべきケース

コンテナワークロードに特化した環境が必要な場合。Flatcar と同様に軽量で攻撃対象面が小さく、最小限のパッケージ構成を求める場合に最適です。Azure に最適化されたカーネルでパフォーマンスも向上します。

🟠 Ubuntu を選ぶべきケース

幅広いパッケージエコシステムや特定のツール・ライブラリが必要な場合。既存の Ubuntu ベースのスクリプトや設定が流用可能で、チームが Ubuntu に精通している場合に適しています。

⚠️ 移行時の注意事項
🔧

Flatcar 固有の設定の変換

Ignition 設定など Flatcar 固有のカスタマイズがある場合、移行先 OS に対応した設定形式に手動で変換する必要があります。

🏷️

ラベル・taint の引き継ぎ

ノードプールに適用されているラベルや taint を新しいノードプールにも正しく設定してください。

🛡️

PodDisruptionBudget の設定

PodDisruptionBudget を適切に設定し、ワークロードの可用性を維持しながら移行を実施してください。

⏱️

ダウンタイムの可能性

ノードプールの追加・削除に伴い、ダウンタイムが発生する可能性があります。事前にテスト環境で検証することを推奨します。

🧪

動作テストの実施

移行先 OS でのアプリケーション動作テストを本番移行前に必ず実施してください。パッケージの違いにより一部ワークロードの動作確認が必要です。
