各クーラーティア (Cool/Cold/Archive) では 128 KiB 未満のオブジェクトが 128 KiB として課金される。Hot ティアには最小課金サイズが適用されないため、小さなオブジェクトのパッキングまたは Smart Tier の活用がコスト軽減策となる。

主要機能

📏

最小課金オブジェクトサイズ: 128 KiB

Cool、Cold、Archive ティアに格納されたオブジェクトに 128 KiB の最小課金サイズが適用される。128 KiB 未満のオブジェクトは、対応するティアの料金で 128 KiB として課金される。課金は既存の容量課金メーター (データ格納量) を使用し、トランザクション課金には影響しない。

📅

段階的展開スケジュール

第1段階 (2026年7月1日): 新規作成されたストレージアカウントにのみ適用。既存のストレージアカウントには影響なし。第2段階 (2027年7月1日): すべてのストレージアカウントに適用。ストレージアカウントの作成日時で適用段階が決定される。

📊

新しい Blob Capacity メトリクス

BlockBlobSmall (128 KiB 未満のブロック Blob) と Data Lake Storage Small (128 KiB 未満の ADLS オブジェクト) の新しい Blob タイプにより、影響を受けるオブジェクトを Azure Portal の Blob Capacity メトリクスで確認可能。

🔥

Hot ティアは対象外

Hot アクセスティアには最小課金オブジェクトサイズが適用されない。引き続き実サイズで課金されるため、小さなオブジェクトを Hot ティアに保持することで最小課金サイズの影響を回避できる。

ユースケース

📡

IoT テレメトリデータの最適化

数千台の IoT デバイスから小さなテレメトリファイル (数百バイト~数 KiB) が Blob Storage にアップロードされ、ライフサイクルポリシーで Cool ティアに移動されている。
対策: 小さなセンサーデータを日次で Parquet ファイルにパッキングしてから Cool ティアに移動。数千の小さなオブジェクトを 1 つの大きなオブジェクトに結合し、最小課金サイズの影響を排除。
⚙

Smart Tier 活用による自動最適化

さまざまなサイズのオブジェクトが混在するストレージアカウントで、手動でのティア管理やパッキングの運用負荷を避けたい。ZRS/GZRS/RA-GZRS 構成を使用中。
対策: Smart Tier を有効化すると、128 KiB 未満のオブジェクトが自動的に Hot ティアに固定され、最小課金サイズの影響を回避。128 KiB 以上のオブジェクトはアクセスパターンに基づいて自動最適化。
⚙

推奨アクション

即時

Azure Monitor で 128 KiB 未満オブジェクトを調査

Azure Monitor の Blob Capacity メトリクスで、クーラーティアに格納されている 128 KiB 未満のオブジェクト数を確認する。新しい BlockBlobSmall / Data Lake Storage Small メトリクスを活用して影響範囲を精査する。

短期 (~2026/7)

新規アカウント設計に最小課金サイズを考慮

新規ストレージアカウントの作成時には最小課金サイズの影響を考慮した設計を行う。小さなオブジェクトをクーラーティアに移動する前にパッキングする運用フローを組み込む。

中期 (~2027/7)

パッキング導入、Smart Tier 移行、ライフサイクルポリシー見直し

既存のストレージアカウントについて、小さなオブジェクトのパッキング処理 (TAR/ZIP) の導入、Smart Tier への移行、またはライフサイクルポリシーの見直しを実施する。128 KiB 未満のオブジェクトについてはルールから除外するか、事前にパッキングする運用を検討する。

⚠

制限事項・注意点

  • Smart Tier の利用には ZRS / GZRS / RA-GZRS が必須 (LRS / GRS のストレージアカウントでは利用不可)
  • パッキング後の個別ファイル取得にはアンパック処理が必要となり、読み取りの複雑さが増す
  • 全リージョンに適用される (例外なし)
  • 2027年7月1日までにすべてのストレージアカウントが対象となるため、影響を受けるワークロードの事前調査と対策が必須
  • パッキング処理の実装と運用が新たに必要になる場合がある

関連サービス

Smart Tier

2026年4月14日に同時に GA となった自動ティアリング機能。128 KiB 未満のオブジェクトを Hot ティアに自動保持するため、最小課金サイズの影響を回避する手段として有効。

ライフサイクル管理

ルールベースでオブジェクトのティア移動を自動化する機能。小さなオブジェクトの移動ルールの見直しが推奨される。

Blob インベントリレポート

ストレージアカウント内のオブジェクトの一覧とサイズを取得できる機能。影響を受けるオブジェクトの特定に活用可能。

Azure Monitor

新しい BlockBlobSmall / Data Lake Storage Small メトリクスで影響を受けるオブジェクトの分布を可視化できる。
