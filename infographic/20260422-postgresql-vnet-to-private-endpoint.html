Azure Database for PostgreSQL Flexible Server In Preview

VNet 統合から Private Endpoint 対応
ネットワーク構成への移行

既存の VNet 統合 (Private Access) でデプロイされた PostgreSQL Flexible Server を、サーバー再作成不要で Private Endpoint 対応構成にインプレース移行できる新機能

📅 2026年4月22日 リリース

概要

Azure Database for PostgreSQL Flexible Server において、仮想ネットワーク (VNet) 統合でデプロイされたサーバーを、Private Endpoint 接続をサポートするネットワーク構成に移行できる機能がパブリックプレビューとして提供開始されました。

これまで VNet 統合モードでデプロイされたサーバーは、ネットワーキングモードを変更できず、Private Endpoint を使用するにはサーバーの再作成とデータ移行が必要でした。今回のアップデートにより、既存サーバーをそのまま Private Endpoint 対応構成に直接移行できるようになりました。

❌ Before

  • VNet 統合サーバーは Private Endpoint を構成不可
  • モード変更にはサーバー再作成とデータ移行が必要
  • デリゲートされたサブネットが PostgreSQL 専用で制約大
  • 別の VNet やサブネットへの移動が不可能

✅ After

  • 既存サーバーを Private Endpoint 構成に直接移行可能
  • サーバー再作成・データ移行が不要に
  • ピアリング VNet やオンプレミスからの接続が可能
  • 異なるリージョンからの Private Link 接続に対応

主要機能

🔄

インプレース移行

既存のサーバーを再作成することなく、VNet 統合 (Private Access) からPrivate Endpoint 対応構成へネットワーキングモードを変更可能。データ移行やダウンタイムを最小限に抑えた移行パスを提供します。

🌐

柔軟なネットワーク接続性

同一 VNet、ピアリング VNet (同一リージョン・クロスリージョン)、VNet 間接続 (VPN Gateway)、オンプレミス (ExpressRoute/VPN) からの接続に対応。異なるリージョンからのプライベート接続 (グローバルリーチ) もサポートします。

🛡️

データ漏洩防止

Private Endpoint はサービス全体ではなく特定のリソースインスタンスにマッピングされるため、他のリソースへのアクセスがブロックされ、データ漏洩リスクを基本的に防止します。

🔒

ネットワークポリシー対応

Private Endpoint ではネットワークセキュリティグループ (NSG)、ユーザー定義ルート (UDR)、アプリケーションセキュリティグループ (ASG) によるきめ細かなトラフィック制御が可能です。

🔥

ファイアウォールルールとの併用

Private Endpoint とパブリックアクセス/ファイアウォールルールの同時利用が可能。段階的な移行や、特定のシナリオでのパブリックアクセス維持に対応できます。

メリット

💼 ビジネス面

  • 移行コストの削減: サーバー再作成やデータ移行が不要になり、運用工数とリスクを削減
  • ダウンタイムの最小化: インプレース移行により、サービス停止時間を最小限に抑制
  • コンプライアンス対応の強化: Private Link によるデータ漏洩防止機能で、セキュリティ要件への対応が容易に

⚙️ 技術面

  • 柔軟なネットワーク接続: ピアリング VNet やオンプレミスからの接続が可能
  • グローバルリーチ: 異なるリージョンから Private Link 経由で接続可能
  • ネットワークポリシー: NSG、UDR、ASG による細かなトラフィック制御
  • Hub-and-Spoke 親和性: Private DNS ゾーンの一元管理が可能
  • サブネット制約の解消: デリゲートされたサブネットの専用化が不要に

ユースケース

1

ハブアンドスポーク ネットワーク構成への統合

企業がハブアンドスポーク型のネットワークアーキテクチャを採用しており、VNet 統合でデプロイ済みの PostgreSQL サーバーを中央集約型のネットワーク管理に統合したい場合。

Private Endpoint に移行することで、ハブ VNet に Private DNS ゾーンを集中配置し、複数のスポーク VNet やオンプレミスから一元的にアクセス管理が可能になる。
2

マルチリージョン接続の実現

異なるリージョンに展開されたアプリケーションから、単一の PostgreSQL サーバーにプライベート接続したい場合。VNet 統合では同一 VNet 内からのアクセスに限定されていた。

Private Endpoint のグローバルリーチ機能により、異なるリージョンの VNet からも Private Link 経由でセキュアに接続できるようになる。
3

オンプレミスからのハイブリッド接続

オンプレミスのアプリケーションから ExpressRoute または VPN 経由で PostgreSQL サーバーにプライベート接続したい場合。

Private Endpoint を経由することで、オンプレミスから Azure バックボーンネットワークを通じたセキュアな接続が実現される。

制約事項・注意点

  • パブリックプレビュー段階のため、本番ワークロードでの利用は慎重に検討する必要がある
  • 移行後は VNet 統合構成には戻せない可能性がある (公式ドキュメントで要確認)
  • Private Endpoint の数は Azure ネットワーキングの制約 (VNet 内のサブネットにインジェクション可能な Private Endpoint 数) に依存する
  • DNS 構成の変更が必要: VNet 統合時の DNS ゾーンから privatelink.postgres.database.azure.com への移行
  • Private Endpoint 利用時は Private DNS ゾーンの適切な構成とリンクが必要
  • 既存アプリケーションの接続文字列やネットワーク構成の更新が必要になる場合がある

料金情報

Private Endpoint の料金は Azure Private Link の料金体系に基づきます。PostgreSQL Flexible Server 自体の料金に加え、Private Endpoint のリソース料金が発生します。
