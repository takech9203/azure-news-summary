Azure Elastic SAN

Capacity Autoscaling (容量自動スケーリング) が一般提供開始

使用量に基づく自動容量拡張で過剰プロビジョニングと手動管理から解放

Launched (GA) 2026-04-23

概要

Azure Elastic SAN の Capacity Autoscaling 機能が一般提供 (GA) となった。Elastic SAN の容量を使用量に基づいて自動的に拡張するポリシーを設定でき、過剰プロビジョニングや手動での容量管理から解放される。特にボリュームスナップショットを使用する環境で有用であり、スナップショットの蓄積による予期せぬ容量不足を自動的に防止する。

Key Points

  • 未使用容量がしきい値以下になると自動的に容量をスケールアップ
  • 追加容量 (Additional Capacity) のみが自動拡張の対象
  • 最大容量の上限設定により予期しないコスト増加を防止
  • SAN 作成時にも既存 SAN にも設定可能

Before

  • ストレージ容量の管理が手動であり、使用量の監視と手動での容量追加が必要
  • 容量不足を避けるために過剰プロビジョニングが発生しやすく、コスト効率が低下
  • ボリュームスナップショットの蓄積により予期せず容量が不足するリスク
  • ストレージ容量の拡張タイミングを見極めるための監視・アラート設定が運用負荷

After

  • オートスケールポリシーにより、使用量に応じた容量の自動拡張が可能に
  • 過剰プロビジョニングを回避し、実際の使用量に近い容量で運用しコスト最適化
  • スナップショット蓄積による容量不足を自動的に防止し運用リスク低減
  • 最大容量の上限を設定でき、予期しないコスト増加を防止

📈

使用量ベースの自動容量拡張

未使用容量がしきい値 (UnusedSizeTiB) 以下になると自動的にスケールアップ。最小増分は 1 TiB で、きめ細かな容量管理が可能。

⚙

追加容量 (Additional Capacity) のみの自動拡張

ベース容量 (IOPS / スループットを決定する部分) は変更されず、追加容量のみが自動拡張の対象。パフォーマンス特性を維持しつつ容量のみを拡張する設計。

🔧

SAN 作成時または既存 SAN への後からの有効化

新規 Elastic SAN 作成時にオートスケールポリシーを設定することも、既存の SAN に対して後から有効化することも可能。柔軟な導入が可能。

Code Samples

az elastic-san create \
  -n myElasticSan \
  -g myResourceGroup \
  -l eastus \
  --base-size-tib 100 \
  --extended-capacity-size-tib 20 \
  --sku "{name:Premium_LRS,tier:Premium}" \
  --availability-zones 1 \
  --auto-scale-policy-enforcement Enabled \
  --unused-size-tib 20 \
  --increase-capacity-unit-by-tib 5 \
  --capacity-unit-scale-up-limit 150
New-AzElasticSAN `
  -ResourceGroupName "myResourceGroup" `
  -Name "myElasticSan" `
  -AvailabilityZone 1 `
  -Location "eastus" `
  -BaseSizeTib 100 `
  -ExtendedCapacitySizeTiB 20 `
  -SkuName Premium_LRS `
  -AutoScalePolicyEnforcement "Enabled" `
  -UnusedSizeTiB 20 `
  -IncreaseCapacityUnitByTiB 5 `
  -CapacityUnitScaleUpLimit 150 `
  -PublicNetworkAccess Disabled

Use Cases

UC1: ボリュームスナップショット管理の自動化

ボリュームスナップショットを定期的に取得する環境では、スナップショットの蓄積により SAN の容量が予期せず不足するリスクがある。Capacity Autoscaling により、スナップショットの増加に応じて容量が自動拡張され、運用担当者が容量不足を心配する必要がなくなる。

-- スナップショットの蓄積による予期せぬ容量不足を防止し、運用負荷を大幅に低減。

UC2: 成長するデータベースワークロード

AKS 上のデータベースワークロードなど、データ量が継続的に増加する環境において、ストレージ容量の手動管理は運用負荷が高い。Capacity Autoscaling を設定することで、データ量の増加に自動的に追従し、容量不足によるサービス中断を防止する。

-- AKS 上の DB ワークロードのデータ量増加に自動追従し、サービス継続性を確保。

Limitations / Caveats

  • ベース容量 (IOPS・スループット) は自動拡張されない -- 追加容量のみが対象
  • スケールダウン (容量の自動縮小) はサポートされていない
  • 最大容量の上限はリージョンと冗長性タイプによるスケールターゲットに制限される
