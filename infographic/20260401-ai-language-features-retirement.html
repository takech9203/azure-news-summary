Azure AI Language: 8 つのレガシー機能のリタイア発表

Azure AI Language (Foundry Tools) の 8 つの機能が 2029 年 3 月 31 日にサポート終了。Key Phrase Extraction、Sentiment Analysis、CLU、CQA など主要な NLU 機能が対象。Microsoft Foundry モデルおよびコア機能への移行が必要です。

🔄 概要: Before / After 比較

Azure AI Language の 8 つの機能が「Legacy capabilities」として分類され、2029 年 3 月 31 日にリタイアします。新規開発にはコア機能または Microsoft Foundry モデルへの移行が推奨されます。

Before: リタイア前の状態

  • 8 つの機能が Azure AI Language の一部として利用可能
  • REST API、クライアントライブラリ (C#, Java, JS, Python) で提供
  • Docker コンテナでオンプレミス展開可能
  • Language Studio (現 Foundry) で GUI 利用可能

After: リタイア後の変更

  • 2029/3/31 以降、対象機能のサポート終了
  • サポート終了後は機能の利用が不可能に
  • Foundry モデルまたはコア機能への移行が必要
  • コア機能 (PII, NER, TAH) は継続利用可能
重要: LUIS から CLU、QnA Maker から CQA へ移行済みのユーザーは、再度 Foundry エージェントへの移行が必要です（二重移行）。早期の移行計画策定を推奨します。

事前構築済み機能 (Pre-built)

🔑

Key Phrase Extraction → Microsoft Foundry モデル

キーフレーズ抽出機能。Foundry モデルのプロンプトベースのアプローチに移行します。

💬

Sentiment Analysis & Opinion Mining → Microsoft Foundry モデル

感情分析・意見マイニング機能。Foundry モデルで同等以上の分析が可能です。

🔗

Entity Linking → Named Entity Recognition (NER) / Foundry モデル

エンティティリンキング機能。コア機能の NER または Foundry モデルに移行します。

🌐

Language Detection → コア機能として継続

言語検出機能はコア機能として継続提供されます。移行の影響は最小限です。

カスタム機能 (Custom)

📄

Custom Text Classification → Microsoft Foundry モデル

カスタムテキスト分類機能。Foundry モデルのファインチューニングまたはプロンプトベースの分類に移行します。

🤖

CLU → Azure Language Intent Routing Agent (Foundry)

会話言語理解機能。Foundry の Intent Routing Agent に移行。CLU は LUIS からの移行先でもあったため、二重移行となるケースがあります。

❓

CQA → Azure Language Exact Question Answering Agent (Foundry)

カスタム質問応答機能。Foundry の Exact QA Agent に移行。CQA は QnA Maker からの移行先でもあったため、二重移行となるケースがあります。

🔄

Orchestration Workflow → Azure Language Intent Routing Agent (Foundry)

オーケストレーションワークフロー機能。Intent Routing Agent で同等のルーティング機能を実現します。

🛡️

🏷️

🏥

Text Analytics for Health

医療テキスト分析機能。ヘルスケア分野の専門的な NLP として継続提供されます。

🌐

Language Detection (コア版)

言語検出機能のコア版。レガシー版からの移行先としても機能します。

LUIS / QnA Maker からの移行済みユーザーへの影響

過去に LUIS から CLU、QnA Maker から CQA へ移行したユーザーは、再度 Foundry エージェントへの移行が必要となります。短期間で 2 回目の移行が発生するため、早期の計画策定が重要です。

💡 ユースケース別 移行ガイダンス

テキスト分析・感情分析

→ Foundry モデル (プロンプトベース)

Key Phrase Extraction、Sentiment Analysis を利用していたユーザー。Foundry モデルでプロンプトベースの柔軟な分析が可能になります。

会話 AI・チャットボット

→ Intent Routing Agent + Exact QA Agent

CLU や CQA を利用した会話 AI やチャットボット。Foundry エージェントでよりインテリジェントなルーティングと応答が可能になります。

ドキュメント分類

→ Foundry モデル

Custom Text Classification を利用していたユーザー。Foundry モデルで再トレーニングなしにゼロショット/フューショット分類が可能です。

エンティティ認識

→ NER (コア機能) + Foundry モデル

Entity Linking を利用していたユーザー。コア機能の NER が継続利用可能で、追加の抽出には Foundry モデルを組み合わせられます。

⚠️ 制約事項・注意点
🔄

二重移行のコスト

LUIS / QnA Maker から移行済みのユーザーは短期間で再度の移行が必要です。移行工数とコストを早期に見積もってください。

🏗️

アーキテクチャ変更の可能性

Foundry モデルへの移行には API エンドポイントや呼び出しパターンの変更が必要な場合があります。既存のアプリケーション設計の見直しが発生する可能性があります。

📦

Docker コンテナ利用者への影響

オンプレミス環境で Docker コンテナを利用して展開しているユーザーは、アーキテクチャの再設計が必要になります。

🧪

Foundry エージェントのプレビュー状態

Intent Routing Agent や Exact QA Agent はプレビュー段階であり、GA 時に仕様が変更される可能性があります。

🔧

カスタムモデルの再構築

カスタムモデルの再トレーニングや検証に相応の工数が必要です。移行スケジュールに余裕を持たせてください。
