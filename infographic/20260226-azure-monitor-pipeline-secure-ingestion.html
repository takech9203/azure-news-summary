TLS/mTLS によるセキュアインジェストと Kubernetes ポッド配置制御

Public Preview 2026-02-26

概要

Azure Monitor pipeline に対して、外部エンドポイントからのセキュアインジェスト（TLS/mTLS 対応）と、Kubernetes クラスター上でのポッド配置制御という 2 つの主要機能がパブリックプレビューとして発表された。BYOC（Bring Your Own Certificates）機能により、企業が独自の証明書を使用して TLS/mTLS 通信を構成でき、セキュリティとコンプライアンス要件への対応が強化された。

Key Points

  • TLS/mTLS による外部エンドポイントからのセキュアなデータ取り込みを実現
  • BYOC により企業独自の PKI 証明書を使用した通信構成が可能に
  • ポッド配置制御により Kubernetes ノードへのスケジューリングを細かく制御
  • OpenTelemetry エコシステムを基盤とした Arc 対応 K8s クラスター上の統合監視

Before

  • TLS/mTLS を企業独自の PKI 証明書で構成する手段が限定的
  • Pod がデフォルトのスケジューリングに依存し、配置制御が困難
  • マルチテナント環境でのパフォーマンス分離やコンプライアンス適用が困難
  • 同一ノードに複数インスタンスが配置されポート枯渇やリソース競合のリスク

After

  • BYOC により企業独自の証明書・PKI での TLS/mTLS 通信構成が可能
  • constraints / distribution 設定による柔軟なスケジューリング制御
  • maxInstancesPerHost でノードあたりのインスタンス数を厳密制限
  • 自動証明書管理とゼロダウンタイムローテーションも引き続き利用可能

🔒

Secure Ingestion (TLS/mTLS)

外部エンドポイントからの TCP ベースレシーバーに対する TLS / mutual TLS (mTLS) サポート。3 つの TLS モード (mutualTls / serverOnly / disabled) から選択可能。mTLS ではサーバーとクライアント双方の証明書認証を実施。

📜

Bring Your Own Certificates (BYOC)

企業独自の証明書と秘密鍵をパイプラインの TLS エンドポイントに使用可能。Azure Key Vault との統合 (Secret Store Extension 経由) や外部 PKI (LetsEncrypt 等) との連携もサポート。

🔄

Automated Certificate Management

cert-manager 拡張機能による自動化された証明書ライフサイクル管理。サーバー証明書は 48 時間の有効期限で、24 時間前に自動更新。ゼロダウンタイムでの証明書ローテーション。

🎯

Pod Placement Control

executionPlacement プロパティにより Kubernetes ノードへのスケジューリングを制御。constraints でノードラベル / AZ / インスタンスタイプに基づく配置制約を設定。distribution の maxInstancesPerHost でノードあたりのインスタンス数を制限。

🌐

Gateway Integration (Traefik)

Traefik Gateway を使用した外部クライアントへのパイプラインエンドポイント公開。Gateway とパイプライン間の mTLS 通信。cert-manager による Gateway クライアント証明書の自動発行・更新。

BYOC TLS Configuration (ARM Template)

{
  "type": "Microsoft.Monitor/pipelineGroups",
  "apiVersion": "2025-03-01-preview",
  "name": "byoc-pipeline",
  "location": "eastus2",
  "properties": {
    "receivers": [
      {
        "name": "syslog",
        "type": "Syslog",
        "tlsConfiguration": "byoc-mtls",
        "syslog": { "endpoint": "0.0.0.0:514" }
      }
    ],
    "tlsConfigurations": [
      {
        "name": "byoc-mtls",
        "mode": "mutualTls",
        "tlsCertificate": {
          "certificate": {
            "type": "kubernetesSecret",
            "location": "collector-server-tls",
            "subLocation": "tls.crt"
          },
          "privateKey": {
            "type": "kubernetesSecret",
            "location": "collector-server-tls",
            "subLocation": "tls.key"
          }
        },
        "clientCa": {
          "type": "kubernetesSecret",
          "location": "byoc-client-root-ca-secret",
          "subLocation": "ca.crt"
        }
      }
    ]
  }
}

Pod Placement Configuration

{
  "properties": {

        {
          "capability": "team",

          "operator": "In",
          "values": ["us-east-1a", "us-east-1b"]

      ],

        "maxInstancesPerHost": 1
      }
    }

Benefits

Business

  • BYOC により企業 PKI ポリシーを維持したまま Azure Monitor pipeline を利用可能
  • ポッド配置 constraints で特定 AZ でのデータ処理を強制、データ主権要件に対応
  • 既存の証明書管理インフラとの統合でセキュリティポリシーに準拠
  • 自動証明書管理で運用チームの作業負荷を削減

Technical

  • TLS/mTLS によるエンドツーエンドの通信暗号化とクライアント認証
  • 自動管理 (cert-manager) と BYOC の 2 つのオプションから選択可能
  • ゼロダウンタイムでの証明書ローテーション
  • maxInstancesPerHost: 1 によるノード単位の厳密リソース分離
  • Azure Key Vault 統合による証明書の安全な保管と自動同期

Use Cases

UC1: Regulated Industry - Secure Log Collection

金融機関がオンプレミスのネットワーク機器から Syslog データを Azure Monitor に取り込む際、社内の PKI 証明書を使用した mTLS 通信を要求されるケース。

Implementation:
  • 社内 PKI 発行のサーバー証明書とクライアント CA を Kubernetes Secret として作成
  • BYOC mTLS 構成でパイプラインの Syslog レシーバーを設定
  • Traefik Gateway 経由で外部ネットワーク機器からの接続を受付
  • ポッド配置で金融データ処理専用ノードへの配置を強制

-- 規制要件を満たしたセキュアな通信経路でログデータを収集し、監査要件にも対応可能。

UC2: Multi-Tenant K8s Cluster Isolation

複数のチームが共有する Kubernetes クラスター上で、各チームの監視パイプラインを適切に分離して運用するケース。

Implementation:
  • constraints でチーム専用ノード (team=observability-team) への配置を指定
  • maxInstancesPerHost: 1 でノードあたり 1 インスタンスに制限
  • チームごとに異なる TLS 証明書を使用

-- ノイジーネイバー問題を回避し、チーム間のパフォーマンス / セキュリティ分離を実現。

UC3: High-Availability Telemetry Collection

大規模な IoT/OT 環境で、高スループットのテレメトリデータを複数の AZ に分散して処理するケース。

Implementation:
  • ポッド配置で特定の AZ (us-east-1a, us-east-1b) に制約
  • 高リソースノード (Standard_D16s_v3 等) をターゲットに設定
  • maxInstancesPerHost: 1 でノード間の負荷分散を確保
  • ローカルキャッシュを活用した断続的接続環境への対応

-- ゾーン冗長性を確保しつつ、高スループットのテレメトリデータを効率的に処理。

Limitations / Caveats

  • Public Preview -- SLA 対象外。本番環境への導入は GA 後を推奨
  • サポートされる K8s ディストリビューションが限定的 (Canonical, Cluster API Provider for Azure, K3, RKE, VMware TKG)
  • BYOC 証明書のローテーションは企業側で管理する必要がある
  • maxInstancesPerHost は現在 1 のみサポート
  • ポッド配置は ARM/Bicep テンプレートのみ対応 (Azure Portal 非対応)
  • cert-manager 拡張機能のインストールが必須 (既存 cert-manager は事前にアンインストールが必要)
