Azure Blob Storage GA 2026-06-30

クライアント側データ整合性保護
(CRC64-NVME SDK 統合)

Azure Blob Storage のクライアント側データ整合性保護が GA。CRC64-NVME チェックサムが最新の Azure Blob SDK (.NET、C++、JavaScript) に統合され、アップロードおよびダウンロード操作時のデータ整合性を自動的に検証可能に。

📋 概要
SDK は Structured Body Format でデータをセグメント化し、各セグメントに CRC64-NVME チェックサムを埋め込むことで、ストリーミング転送中でもインクリメンタルに整合性を検証。開発者は設定を変更するだけで自動的にデータ保護を有効化できる。

Before

  • CRC64-NVME チェックサムを利用するには、REST API レベルでヘッダーを明示的に設定する必要があった
  • MD5 ハッシュは計算コストが高く、大容量データでのパフォーマンスに影響があった
  • ブロック単位のアップロードでは各ブロックの整合性検証を個別に実装する必要があった

After

  • SDK が自動的に CRC64-NVME チェックサムを計算・検証するため、開発者の実装負担が大幅に軽減
  • Structured Body Format により、ストリーミング転送中でもセグメント単位で整合性を検証可能
  • StorageChecksumAlgorithm.Auto 設定で SDK が最適なアルゴリズムを自動選択

クライアント SDK が CRC64-NVME チェックサムを計算し、Structured Body Format で転送。サーバー側で検証し不一致の場合は 400 エラーを返却。

⭐ 主要機能
🛡

自動チェックサム計算と検証

SDK がアップロード・ダウンロード時に自動的に CRC64-NVME チェックサムを計算。サーバー側でも検証し、不一致の場合は 400 エラーを返却してデータ破損を即座に検出。

📦

Structured Body Format

データを 4MiB セグメントに分割し、各セグメントにチェックサムを付与。大容量ファイルでもインクリメンタルに整合性を確認でき、転送途中の破損も即座に検出可能。

🔧

柔軟なアルゴリズム選択

Auto / StorageCrc64 / MD5 / None から選択可能。Auto 設定では SDK がデータサイズや環境に応じて最適なアルゴリズムを自動選択。

⚙

クライアント・メソッドレベルでの設定

BlobClientOptions でクライアント全体にデフォルト設定を適用し、個別のメソッド呼び出しでオーバーライド可能。きめ細かな制御を実現。

💻 コードサンプル

C# - CRC64-NVME チェックサムによるアップロード

var validationOptions = new UploadTransferValidationOptions
{
    ChecksumAlgorithm = StorageChecksumAlgorithm.Auto
};

var uploadOptions = new BlobUploadOptions()
{
    TransferValidation = validationOptions
};

BlobClient blobClient = containerClient.GetBlobClient("myblob.dat");
await blobClient.UploadAsync(fileStream, uploadOptions);
💡 ユースケース

🎬 大容量メディアファイルのアップロード

映像制作会社が数 GB のメディアファイルをクラウドストレージにアップロードする際、ネットワーク経路上でのビット反転やパケットロスによるデータ破損を確実に検出する必要がある。

効果: CRC64-NVME によるセグメント単位の整合性検証で、大容量ファイルでも転送中のデータ破損を即座に検出・リトライ可能。MD5 より高速で計算コストを削減。

📱 IoT/エッジデバイスからのデータ収集

工場の IoT センサーやエッジデバイスから Azure Blob Storage にデータを送信する際、不安定なネットワーク環境でのデータ完全性を保証する必要がある。

効果: SDK の自動チェックサム機能により、不安定なネットワーク環境でもデータの完全性を透過的に保証。開発者は整合性検証のロジックを個別実装する必要がなくなる。
