Prompt Flow 廃止のお知らせ

後継の Microsoft Agent Framework へ移行 -- YAML ベースからコードファーストへのパラダイムシフト

Retirement 2026-04-27

概要

Microsoft は、Azure Machine Learning および Microsoft Foundry における Prompt Flow の廃止を正式に発表した。Prompt Flow の新機能開発は 2026 年 4 月 20 日に終了しており、2027 年 4 月 20 日に完全廃止（読み取り専用モードへの移行）が予定されている。後継となる Microsoft Agent Framework は、コードファーストの Python / .NET フレームワークとして、型安全なワークフロー、組み込みエージェントサポート、ネイティブ OpenTelemetry トレーシング、マルチエージェントオーケストレーションなどの機能を提供する。

Key Points

  • Prompt Flow の新機能開発は 2026 年 4 月 20 日に終了済み
  • 2027 年 4 月 20 日に完全廃止（読み取り専用モード移行）
  • 後継は Microsoft Agent Framework（Python / .NET）
  • WorkflowBuilder による型安全なワークフロー定義が可能に
  • ネイティブ OpenTelemetry トレーシングを標準サポート
  • マルチエージェントオーケストレーションのネイティブサポート

Before (Prompt Flow)

  • YAML ベースのビジュアルグラフ定義に依存し、複雑なワークフロー管理に制約
  • 型安全性の欠如により、エラー検出が実行時に限定
  • マルチエージェントオーケストレーションが未サポート
  • Azure ML 固有の実行履歴に依存し、標準的なオブザーバビリティ統合が限定的

After (Agent Framework)

  • コードファーストの型安全なワークフロー定義（WorkflowBuilder）
  • ビルド時に型の不一致や到達不能ノードを検出可能
  • add_edge() / add_fan_out_edges() でマルチエージェント連携
  • ネイティブ OpenTelemetry + Application Insights 統合

Microsoft Agent Framework の主要機能

🏗

WorkflowBuilder による型安全なワークフロー

コードファーストの Python / .NET でワークフローを定義。ビルド時に型の不一致や到達不能ノードを検出し、ランタイムエラーを未然に防止する。

🤖

組み込みエージェントサポート

AzureOpenAIChatClient().as_agent() で LLM をエージェントとして利用可能。Python 関数をツールとして簡単に登録でき、エージェントの構築が容易。

🔍

ネイティブ OpenTelemetry トレーシング

configure_azure_monitor() 一行で Application Insights への接続が完了。標準的なオブザーバビリティスタックとシームレスに統合。

🔗

マルチエージェントオーケストレーション

add_edge()、add_fan_out_edges()、add_fan_in_edges() により、複数エージェントの連携・分岐・合流をネイティブにサポート。

🚀

柔軟なデプロイメント

Azure Container Apps、Azure Functions など標準的なコンテナホストへデプロイ可能。FastAPI ラッパーで REST API として公開。

移行 5 ステップ

移行は以下の 5 フェーズで段階的に進める。各フェーズにはゲート（進行判定基準）があり、順番に完了させる。

1

監査とマッピング

pf flow export で既存フローをエクスポートし、flow.dag.yaml の全ノードを Agent Framework の対応コンポーネントにマッピング。

Gate: ノードマッピングテーブルがフロー内の全ノードを網羅
2

再構築

WorkflowBuilder と Executor クラスを使用してワークフローを再実装。まず単一の線形パスから始め、次に分岐・ファンアウト・条件付きエッジを追加。

Gate: Python ファイルが Prompt Flow の動作を再現する出力を生成
3

検証

元の Prompt Flow と新しい Agent Framework を同一入力で並行実行。Azure AI Evaluation SDK の SimilarityEvaluator で出力の同等性をスコアリング。

Gate: 平均類似度スコア 3.5 以上（5 段階中）
4

運用移行

OpenTelemetry トレーシングを設定し、Azure Container Apps にデプロイ。CI/CD パイプラインに同等性評価を品質ゲートとして組み込み。

Gate: Application Insights にトレースが表示され、デプロイ正常
5

切り替え

本番トラフィックを Agent Framework デプロイメントに切り替え。安定稼働確認後、Prompt Flow リソースを廃止。

Gate: 本番トラフィックの安定稼働を確認

検証スコア判定基準

スコア範囲 意味 アクション
3.5 未満 出力が大きく乖離 プロンプトコンテキストの不足や未移行ノードを確認
3.5 - 4.5 軽微な表現差異 一般的に許容可能
4.5 超 高い意味的一致 デプロイメントに進んでよい

Code Samples

パッケージインストール

pip install agent-framework>=1.0.0 azure-ai-evaluation pandas python-dotenv

# RAG ワークフローの場合
pip install agent-framework-azure-ai-search

WorkflowBuilder によるワークフロー構築例

from agent_framework import Executor, WorkflowBuilder, WorkflowContext, handler
from agent_framework.azure import AzureOpenAIChatClient

workflow = (
    WorkflowBuilder(name="MyWorkflow")
    .register_executor(lambda: InputExecutor(id="input"), name="Input")
    .register_executor(lambda: LLMExecutor(id="llm"), name="LLM")
    .add_edge("Input", "LLM")
    .set_start_executor("Input")
    .build()
)

OpenTelemetry トレーシングの設定

from azure.monitor.opentelemetry import configure_azure_monitor

configure_azure_monitor(
    connection_string=os.environ["APPLICATIONINSIGHTS_CONNECTION_STRING"],
)

Use Cases

UC1: LLM ベースの Q&A アプリケーションの移行

Prompt Flow で構築した入力ノード + LLM ノードの線形フローを Microsoft Agent Framework に移行。

-- WorkflowBuilder と AzureOpenAIChatClient().as_agent() による型安全な実装に切り替わり、OpenTelemetry トレーシングによる運用監視が可能に。

UC2: RAG パイプラインの統合移行

Prompt Flow の Embed Text ノード、Vector DB Lookup ノード、LLM ノードの 3 つに分かれていた RAG パイプラインを移行。

-- AzureAISearchContextProvider により 3 ノードが 1 構成に統合され、エンベディングとベクトル検索が自動処理。

UC3: マルチエージェントカスタマーサポートの構築

Prompt Flow の条件分岐ノードで実現していた分類 + 専門エージェントルーティングを、Agent Framework のマルチエージェントハンドオフパターンに移行。

-- トリアージ、請求担当、技術担当エージェントの連携が add_edge() の条件関数で明確に定義され、保守性向上。

Limitations / Caveats

  • Agent Framework にはビジュアルグラフエディタが含まれていないため、コードベースのワークフロー定義への移行計画が必要
  • 既存の Prompt Flow ワークロードはすべて手動で再実装する必要があり、移行工数が発生
  • マネージドオンラインエンドポイントの直接的な代替がなく、FastAPI ラッパー + コンテナデプロイへの変更が必要
  • agent-framework-azure-ai-search パッケージは現時点でプレビュー段階
  • ワークスペース接続による資格情報管理が環境変数ベースに変更となるため、シークレット管理の見直しが必要

関連サービス

Microsoft Foundry Agent Service

エージェントの構築、デプロイ、スケーリングを行うフルマネージドプラットフォーム。Prompt / Workflow / Hosted の 3 種類対応。

Azure OpenAI Service

Agent Framework が LLM バックエンドとして利用。AzureOpenAIChatClient で接続。

Azure AI Search

RAG ワークフローにおけるベクトル検索基盤。AzureAISearchContextProvider 経由で統合。

Azure Container Apps

Agent Framework ワークフローの推奨デプロイ先。スケーラブルなコンテナホスティング。

Azure AI Evaluation SDK

移行時の出力同等性検証ツール。SimilarityEvaluator / CoherenceEvaluator 等。

Application Insights

Agent Framework の OpenTelemetry トレーシング送信先。運用監視の中核。
