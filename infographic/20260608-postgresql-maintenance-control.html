概要

Azure Database for PostgreSQL Flexible Server で、メンテナンスのリスケジュール (最大 14 日延期)、オンデマンド適用 (即時実行)、メンテナンス履歴の表示・CSV ダウンロードが GA。ビジネス影響の最小化と運用の可視化を実現。

❌ Before

  • ・スケジュール後に日時変更手段なし
  • ・即時適用したくてもスケジュール待ち
  • ・過去のメンテナンス履歴の確認手段なし
  • ・メンテナンス詳細を一覧で把握しにくい

✅ After

  • ・最大 14 日間先まで延期可能
  • ・オンデマンドで即時適用可能
  • ・Azure Portal で履歴確認 + CSV エクスポート
  • ・種類、ステータス、リスケジュール可否を事前確認

主要機能

📅

メンテナンスのリスケジュール

予定されたメンテナンスを最大 14 日間延期可能。ピーク時間、リリースウィンドウ、決算期など、ビジネスクリティカルな期間を回避。複数回の変更も可能。

⚡

オンデマンドメンテナンス適用

スケジュールを待たずにメンテナンスを即時適用。低負荷時間帯を選んで自発的に実行可能。「Scheduled」「Rescheduled」状態の場合に利用可能。

📋

今後のメンテナンス表示

Overview の「Next Maintenance」フィールドで次回予定を確認。メンテナンスページで詳細情報 (種類、ステータス、リスケジュール可否) を表示。

📥

メンテナンス履歴の表示・ダウンロード

過去のメンテナンスイベント一覧を表示。種類、開始・終了時間、最終ステータスを確認し、CSV エクスポートで監査対応も可能。

設定例

カスタムメンテナンスウィンドウの設定 (Azure CLI)

Bash 
# カスタムスケジュールの設定 (水曜日 14:29 UTC 開始)
az postgres flexible-server update \
  --resource-group <resource_group> \
  --name <server> \
  --maintenance-window Wed:14:29

# システム管理スケジュールに戻す
az postgres flexible-server update \
  --resource-group <resource_group> \
  --name <server> \
  --maintenance-window Disabled

ユースケース

🛍️ EC サイトの大規模セール期間のメンテナンス回避

年末セール期間中にメンテナンスが予定された場合、Azure Portal からリスケジュールしてセール終了後の低負荷時間帯に変更。売上への影響をゼロに抑える。

🚀 計画的リリースとの整合性確保

アプリのデプロイ直後にメンテナンスが予定されている場合、デプロイの安定確認後に「Apply now」で即時適用。問題切り分けが容易に。

📊 監査・コンプライアンス対応

定期監査で DB メンテナンス実績を報告する必要がある場合、「Export to CSV」で履歴をダウンロードし監査レポートに添付。対応工数を削減。

制限事項・注意点

  • Burstable ティアはリスケジュール非対応 (開発・テスト環境で多用されるティア)
  • 現時点では Azure Portal のみ対応 (CLI / REST API は開発中)
  • リスケジュール上限 14 日 (無期限延期は不可)
  • 緊急メンテナンス (重大脆弱性対応等) はリスケジュール不可
  • カスタムメンテナンスウィンドウの構成が前提条件
  • 予定時間の 15 分前からリスケジュール不可 (ロック期間)
  • メンテナンス実行中は設定変更・起動/停止を避ける必要あり
