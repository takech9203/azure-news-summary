📋 概要
Azure Storage Mover が Azure Government (US) リージョンで一般提供 (GA) を開始。FedRAMP High、DoD Impact Level 4/5 などの米国政府向けコンプライアンス要件を満たす環境で、SMB、NFS、AWS S3 からのフルマネージド移行が可能に。移行データはエージェントからターゲットに直接転送され、差分コピーによるダウンタイム最小化にも対応。

Before

  • Azure Government 環境へのフルマネージド移行サービスが利用不可
  • AzCopy やカスタムスクリプトなど手動ツールに依存し、大規模移行の管理・監視が困難
  • コンプライアンス要件を満たしつつリトライ処理やダウンタイム最小化を自動化する手段が限定的
  • 複数のソース共有を一元管理し、移行プロジェクト単位で進捗を追跡する仕組みがなかった

After

  • Azure Government (US) でフルマネージドの Storage Mover が利用可能、政府向けコンプライアンスに準拠
  • 移行プロジェクト、ジョブ定義、エンドポイントをクラウドから一元管理し可視性と制御性が向上
  • SMB、NFS、AWS S3 の複数プロトコルを単一サービスでカバーし、移行ツールを統一
  • 差分コピーでダウンタイムを最小化しながら段階的な移行を実行可能

Storage Mover エージェント VM がソースストレージからデータを読み取り、Azure Government 内のターゲットに直接転送。制御メッセージとメタデータのみが Storage Mover リソースとやり取りされる。

⭐ 主要機能
🏛

Azure Government (US) での一般提供

FedRAMP High などの政府向けコンプライアンス基準に準拠した Azure Government 環境でフルマネージドの移行サービスを利用可能。米国政府機関のクラウド移行を加速する。

🔄

複数ソースプロトコルのサポート

SMB 2.x/3.x マウント、NFS 3/4 マウント、AWS S3 バケットからの移行に対応。Windows/Linux ファイルサーバーおよびクラウドストレージからの移行を単一サービスでカバー。

📂

フルフィデリティ移行

ファイル内容に加え、フォルダー構造、タイムスタンプ、ACL、ファイル属性などのメタデータも保持。SMB ソースから Azure ファイル共有への移行では同等レベルのファイルフィデリティを維持。

⚡

差分コピーとダウンタイム最小化

初回フルコピー後は差分のみを転送。メタデータのみ変更されたファイルはメタデータのみ更新し、ダウンタイムを最小化しながら段階的な移行が可能。

📊

一元的な移行管理

単一の Storage Mover リソースから複数のソース共有の移行を管理。移行プロジェクト、ジョブ定義、エージェントをリソース階層として構造化し、Azure Portal / CLI / PowerShell から一元管理。

🛠 セットアップ手順

前提条件

1

Azure Government サブスクリプション

Azure Government 環境のアクティブなサブスクリプションが必要

2

仮想化ホスト

ソースストレージ近くに Hyper-V または VMware ホストが利用可能であること

3

ネットワーク接続

エージェント VM からインターネットへの送信 HTTPS (TCP 443) 接続が可能であること

4

リソースプロバイダー登録

Microsoft.StorageMover および Microsoft.HybridCompute リソースプロバイダーの登録が必要

Azure CLI によるセットアップ

Bash 
# リソースプロバイダーの登録
az provider register --namespace Microsoft.StorageMover
az provider register --namespace Microsoft.HybridCompute

# Storage Mover リソースの作成
az storage-mover create \
  --name <storage-mover-name> \
  --resource-group <resource-group-name> \
  --location <azure-government-region>

# 移行プロジェクトの作成
az storage-mover project create \
  --name <project-name> \
  --storage-mover-name <storage-mover-name> \
  --resource-group <resource-group-name>

Azure Portal による手順

1

Azure Government Portal にサインインし「Storage Mover」を検索、「+ 作成」を選択

2

サブスクリプション、リソースグループ、リージョンを指定して Storage Mover リソースを作成

3

エージェント VM イメージをダウンロードし、Hyper-V または VMware でデプロイ

4

エージェント VM に SSH 接続し、デフォルトパスワードを変更後、Storage Mover リソースに登録

5

💡 ユースケース

🏢 政府機関のオンプレミスファイルサーバー移行

米国連邦政府機関が、オンプレミスの Windows ファイルサーバー (SMB) から Azure Government の Azure Files へ大規模なファイル共有を移行する。ACL やファイル属性を含むフルフィデリティ移行が求められ、業務への影響を最小限に抑える必要がある。

効果: Storage Mover の差分コピー機能により、初回フルコピー後は変更分のみを転送しダウンタイムを最小化。移行プロジェクト単位で進捗を追跡し Azure Portal から一元管理が可能。

📦 Azure Data Box との組み合わせによる大規模データ移行

数百テラバイトのデータを持つ政府機関が、初回バルク移行に Azure Data Box を使用し、Data Box 輸送中にソース側で発生した変更を Storage Mover のオンラインキャッチアップで差分転送する。

効果: ネットワーク帯域幅を節約しつつ、Data Box 輸送中の変更差分のみを転送することで、最終カットオーバー時のダウンタイムを大幅に短縮。

☁ AWS から Azure Government への移行

AWS GovCloud の S3 バケットに保存されたデータを Azure Government の Blob Storage へ移行する。クロスクラウド移行においてもフルマネージドのサービスで移行の管理と監視を行いたい。

効果: AWS S3 ソースエンドポイントを定義し Azure Government の Blob コンテナーをターゲットに設定することで、クロスクラウド移行を一元管理できる。

Storage Mover サービス利用 無料 (将来的に追加機能への課金の可能性あり)
ターゲット Azure Storage 利用 ストレージトランザクションおよび使用容量に基づく課金
ネットワーク利用 Azure への送信トラフィックに対する標準のネットワーク料金
エージェント VM オンプレミスの仮想化ホスト上で稼働するため Azure VM コストは発生しない
⚠ 制限事項・注意点
⚠ エージェント VM は Hyper-V または VMware 上でのみサポート。Azure VM としてのデプロイはサポートされていない
⚠ SMB 1.x ソースおよび NFS Azure ファイル共有 (ターゲットとして NFS 4.1 のみ対応) には制限がある
⚠ AWS S3 の Glacier および Glacier Deep Archive ストレージクラスからの移行はサポートされていない
⚠ ジョブ定義作成後にソースおよびターゲットのエンドポイント情報を変更不可 (新規ジョブ定義の作成が必要)
⚠ 登録解除したエージェント VM の再登録はサポートされず、新しいエージェント VM のデプロイが必要
⚠ ストレージトランザクション数の事前見積もりが困難。特に小さなファイルが多数ある場合はトランザクションコストが増加する可能性あり
🔗 関連サービス
Azure Blob Storage

Storage Mover のターゲットとして利用可能。FNS (フラット名前空間) および HNS (階層的名前空間) の両方に対応。

Azure Files

SMB および NFS プロトコルの Azure ファイル共有を移行ターゲットとしてサポート。

Azure Data Box

大規模データの初回バルク移行に使用し、Storage Mover と組み合わせてオンラインキャッチアップを実行可能。

Azure Arc

Storage Mover エージェントの管理に使用。エージェント VM は Azure Arc で接続されたサーバーとして管理される。

Azure Monitor

移行のメトリクスとコピーログを Azure Monitor 経由で監視可能。
