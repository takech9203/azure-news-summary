Azure Storage GA Storage Security

User Delegation SAS の Tables, Files, Queues 拡張

ユーザー委任 SAS が全ストレージサービスで利用可能に。アカウントキー依存を排除し、Entra ID ベースの統一セキュリティを実現

Release: 2026-03-31 Service: Azure Tables / Files / Queues API: sv=2025-07-05+

📋 Overview

Azure Storage のユーザー委任 SAS (User Delegation SAS) が、従来の Blob Storage に加えて Azure Tables、Azure Files、Azure Queues でも一般提供 (GA) となった。ユーザー委任 SAS は、ストレージアカウントキーではなく Microsoft Entra ID の資格情報で署名される SAS トークンであり、アカウントキー漏洩リスクを排除する Microsoft 推奨のベストプラクティスである。

Before

  • Table, Queue, Files でユーザー委任 SAS 使用不可
  • アカウントキーベースの SAS に依存、キー漏洩リスクが高い
  • Blob Storage のみユーザー委任 SAS 対応
  • Microsoft 推奨のベストプラクティスに従えなかった

After

  • 全サービス (Blob, Table, Queue, Files) で利用可能
  • Entra ID 資格情報でトークン署名、キー漏洩リスク排除
  • 統一的なセキュリティポリシー適用が可能に
  • 各サービス向け専用 Delegator ロール追加

ユーザー委任 SAS の認証フロー。アプリケーションが Entra ID を通じてユーザー委任キーを取得し、各ストレージサービス向けの SAS トークンを生成する。サービスごとに専用の Delegator ロールが必要。

⚙ Key Features

📊

Azure Tables ユーザー委任 SAS

テーブルデータへのアクセスを Entra ID ベースで制御

  • クエリ、追加、更新、削除操作を制御可能
  • パーティションキー・行キー範囲の指定によるきめ細かいアクセス制御
📁

Azure Files ユーザー委任 SAS

ファイル共有のアクセスをアカウントキー不要で安全に提供

  • 読み取り、書き込み、削除、一覧表示を制御可能
  • ファイルレベル・共有レベルでのスコープ設定
📨

Azure Queues ユーザー委任 SAS

キューメッセージの送受信を Entra ID 認証で保護

  • 読み取り、追加、更新、処理操作を制御可能
  • マイクロサービス間の安全なメッセージングに最適
🔑

サービス別 Delegator ロール

各サービスに専用の RBAC ロールが追加され、最小権限の原則を適用可能

  • Storage Queue Delegator - キュー用委任キー取得権限
  • Storage Table Delegator - テーブル用委任キー取得権限
  • Storage File Delegator - ファイル用委任キー取得権限

💻 Implementation

1. Queue Delegator ロールの割り当て

Bash (Azure CLI) 
az role assignment create \
  --assignee {principal-id} \
  --role "Storage Queue Delegator" \
  --scope /subscriptions/{sub-id}/resourceGroups/{rg}/providers/Microsoft.Storage/storageAccounts/{account}

2. ユーザー委任キーの取得 (REST API)

HTTP 
POST https://{account}.{service}.core.windows.net/?restype=service&comp=userdelegationkey
Authorization: Bearer {OAuth-token}
x-ms-version: 2025-07-05

{service} には blob, table, queue, file のいずれかを指定。各サービスのエンドポイントに対して User Delegation Key を取得する。

💡 Use Cases

1

マイクロサービスアーキテクチャでのキュー連携

サービス間のメッセージングに Queue Storage を利用する際、アカウントキーを各サービスに配布する代わりに、ユーザー委任 SAS でセキュアなメッセージ送受信を実現する。

✅ アカウントキー不要でキュー操作が可能になり、キー漏洩リスクをゼロに。
2

IoT デバイスの Table Storage 書き込み

IoT デバイスが Table Storage にテレメトリデータを書き込む際、パーティションキー範囲を限定したユーザー委任 SAS を発行し、きめ細かいアクセス制御を適用する。

✅ デバイスごとにパーティション範囲を限定し、他デバイスのデータへのアクセスを防止。
3

外部パートナーとのファイル共有

Azure Files のファイル共有に対して、外部パートナーにユーザー委任 SAS を発行し、アカウントキーを開示することなく安全にファイルを共有する。

✅ アカウントキー非開示で安全な外部コラボレーションを実現。
4

Shared Key アクセスの完全無効化

全ストレージサービスでユーザー委任 SAS が利用可能になったことで、ストレージアカウントの Shared Key アクセスを完全に無効化し、Entra ID ベース認証に統一する。

✅ アカウントキーの存在自体をなくし、ゼロトラスト環境を実現。

⚠ Limitations / Caveats

⚠️ API バージョン要件 - sv=2025-07-05 以降が必要。古い SDK やツールでは利用できない場合がある。
⚠️ ストアドアクセスポリシー非対応 - アドホック SAS としてのみ使用可能。ストアドアクセスポリシーとの組み合わせは不可。
⚠️ 委任キー有効期間 - ユーザー委任キーの有効期間は最大 7 日間。長期間のアクセス委任には定期的なキー更新が必要。
⚠️ 管理操作の制限 - コンテナ・キュー・テーブルの作成・削除・一覧表示は不可。データ操作のみサポート。
⚠️ Queue メタデータ書き込み不可 - Queue のメタデータ操作は制限される。
⚠️ SDK 更新が必要な場合あり - 最新の API バージョンをサポートする SDK への更新が必要となる場合がある。

💰 Pricing

ユーザー委任 SAS 自体に追加料金は発生しない。従来の Azure Storage の料金体系がそのまま適用され、標準のストレージトランザクション料金のみが課金される。
