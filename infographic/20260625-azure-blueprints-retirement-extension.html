Azure Blueprints Retirement 2026-06-25

Azure Blueprints: 廃止タイムラインの延長
2027 年 1 月 31 日まで

Azure Blueprints (Preview) の廃止日が 2027 年 1 月 31 日に延期。2026 年 7 月 31 日から段階的に機能制限が開始され、Azure Deployment Stacks + Template Specs への移行が必要。

~6 か月
廃止日の延長期間
4 フェーズ
段階的廃止スケジュール
2 サービス
推奨移行先
追加料金なし
移行先サービスの利用

📋概要: Before / After

❌ Before

  • 廃止日 2026/7/11 で移行期間が不足
  • 段階的移行の指針が不明確
  • エクスポート・移行手順が複雑
  • 大規模環境での計画立案が困難

✅ After

  • 廃止日が 2027/1/31 に約 6 か月延長
  • 4 フェーズの明確な廃止スケジュール
  • 具体的な移行ガイドをドキュメント化
  • 計画的な移行の優先順位付けが可能

📅段階的廃止タイムライン

重要: 2027 年 1 月 31 日以降、エクスポートされていない Blueprint データは永久削除されます。全定義のエクスポートを最優先で実施してください。

🏗️アーキテクチャ: 移行前 → 移行後

Azure Blueprints の機能は、Template Specs (定義のストレージ・バージョン管理) と Deployment Stacks (デプロイ・ライフサイクル管理・ロック) の 2 つのサービスに分割して移行します。

        BP --> BA["🔗 Blueprint 割り当て"]
        BA --> Policy["🛡️ ポリシー割り当て"]
        BA --> RBAC["👤 ロール割り当て"]

        DS --> Policy2["🛡️ ポリシー割り当て"]
        DS --> RBAC2["👤 ロール割り当て"]
        DS --> Resources["📦 マネージドリソース"]
        DS --> Deny["🔒 Deny Settings"]

⚡主要機能: 推奨移行先

🏗️

Azure Deployment Stacks (推奨移行先)

リソースコレクションを単一ユニットとして管理。Deny Settings によるリソース保護 (Blueprint ロックの代替)、ActionOnUnmanage によるライフサイクル管理。リソースグループ・サブスクリプション・管理グループスコープで作成可能。

📋

Template Specs

ARM テンプレートを Azure 上に保存・バージョン管理。Blueprint 定義のストレージ・共有機能の代替。Deployment Stacks と組み合わせて使用。

🔒

Deny Settings (Blueprint ロック代替)

Blueprint ロックよりも柔軟なリソース保護が可能。denyDelete / denyWriteAndDelete モードで Control Plane 操作を制御。

📝

Bicep による IaC

ARM テンプレートのみだった Blueprint より開発体験が大幅向上。Git リポジトリでの管理により PR レビュー・CI/CD パイプラインとの統合が容易。

🔧技術仕様

項目 詳細
影響対象 Azure Blueprints (Preview) を使用しているすべてのサブスクリプション・管理グループ
段階的廃止開始 2026 年 7 月 31 日
完全廃止日 2027 年 1 月 31 日
推奨移行先 Azure Deployment Stacks + Template Specs
Blueprint ロック代替 Deployment Stacks の Deny Settings
必要なツールバージョン Azure PowerShell 12.0.0 以降、Azure CLI 2.61.0 以降

💻実装例: Deployment Stacks の作成

Azure CLI Blueprint 割り当ての代替 
# Deployment Stacks の作成 (Blueprint 割り当ての代替)
az stack sub create \
  --name 'my-governance-stack' \
  --location 'japaneast' \
  --template-file 'main.bicep' \
  --action-on-unmanage 'detachAll' \
  --deny-settings-mode 'denyDelete'
Azure PowerShell サブスクリプションスコープ 
# Deployment Stacks の作成 (サブスクリプションスコープ)
New-AzSubscriptionDeploymentStack `
  -Name "my-governance-stack" `
  -Location "japaneast" `
  -TemplateFile "main.bicep" `
  -ActionOnUnmanage "detachAll" `
  -DenySettingsMode "DenyDelete"

🎯ユースケース

ユースケース 1: 既存 Blueprint 定義の Deployment Stacks への移行

シナリオ: 組織で Blueprint による標準環境デプロイを行っている場合

Azure CLI 移行手順 
# 1. 既存 Blueprint 定義をエクスポート
az stack group export \
  --name 'existing-blueprint-stack' \
  --resource-group 'governance-rg'

# 2. エクスポートした ARM テンプレートを Bicep に変換
az bicep decompile --file exported-template.json

# 3. Deployment Stacks として再デプロイ
az stack sub create \
  --name 'migrated-governance-stack' \
  --location 'japaneast' \
  --template-file 'main.bicep' \
  --action-on-unmanage 'detachAll' \
  --deny-settings-mode 'denyDelete'
効果: Blueprint の定義・割り当て・ロックを Deployment Stacks で再現し、GA サービスによる安定した運用を実現

ユースケース 2: Template Specs を使用した定義の共有

シナリオ: 複数チームで共通のガバナンステンプレートを共有したい場合

Azure CLI Template Spec + Deployment Stack 
# Template Spec の作成
az ts create \
  --name 'standard-landing-zone' \
  --version '1.0.0' \
  --resource-group 'templates-rg' \
  --template-file 'landing-zone.bicep'

# Template Spec を使った Deployment Stack の作成
az stack sub create \
  --name 'team-a-landing-zone' \
  --location 'japaneast' \
  --template-spec '/subscriptions/{sub-id}/resourceGroups/templates-rg/providers/Microsoft.Resources/templateSpecs/standard-landing-zone/versions/1.0.0' \
  --action-on-unmanage 'detachAll' \
  --deny-settings-mode 'denyWriteAndDelete'
効果: テンプレートのバージョン管理・共有を Template Specs で行い、デプロイとリソース保護を Deployment Stacks で実現

✅メリット

ビジネス面

移行期間が約 6 か月延長され計画的な移行が可能

段階的廃止で移行の優先順位付けが容易

業務影響を最小限に抑えた移行計画を立案可能

技術面

Deployment Stacks は GA で Blueprint (Preview) より安定

Deny Settings は Blueprint ロックより柔軟な保護

Bicep + Git で PR レビュー・CI/CD 統合が容易

⚠️制限事項・注意点

  • 2027 年 1 月 31 日以降、エクスポートされていない Blueprint データは永久削除
  • Deployment Stacks のポータル GUI は一部未実装の機能がある
  • Blueprint の「パッケージ + 割り当て」を 1 サービスで完結できず、Template Specs + Deployment Stacks の 2 つが必要
  • Deny Settings は Control Plane のみに適用。Data Plane 操作 (Blob 作成、Secret 追加等) には効かない
