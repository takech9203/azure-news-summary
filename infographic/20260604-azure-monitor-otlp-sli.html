OTLP ネイティブインジェスト & SLI/SLO

OpenTelemetry Collector 経由の OTLP シグナルインジェストと Service Level Indicators が一般提供開始

📅 2026-06-04 GA 🏷️ Azure Monitor

📋 概要

Azure Monitor の観測性基盤を強化する 2 つの重要な GA が発表されました。

  • OTLP ネイティブインジェスト: OpenTelemetry Collector から Azure Monitor クラウドエンドポイントへ直接テレメトリ送信が可能に
  • SLI/SLO 機能: 顧客体験に基づく複合インジケータの定義とサービスレベル目標の追跡が可能に
  • ベンダーニュートラルなテレメトリパイプラインを維持しながら Azure Monitor の分析・可視化機能を活用
  • メトリクス・トレース・ログの 3 種類のシグナルすべてに対応

❌ Before (課題)

  • OTel Collector から Azure Monitor への公式ネイティブ OTLP パスが存在しなかった
  • ベンダーニュートラルパイプラインに追加の統合作業が必要
  • 複数メトリクスを手動で組み合わせた SLI/SLO 管理が困難
  • インフラメトリクスと顧客体験の間にギャップ

✅ After (改善)

  • Collector から Azure Monitor エンドポイントへ OTLP シグナルを直接送信可能
  • Entra ID 認証 + DCR によるセキュアかつ宣言的なパイプライン
  • 複合 SLI 定義で顧客体験ベースのサービスレベル計測
  • Application Insights, Log Analytics, Grafana で統合分析

⚡ 主要機能

📡
OTLP ネイティブインジェスト
OpenTelemetry Collector から Azure Monitor クラウドインジェストエンドポイントへ OTLP シグナルを直接送信。ベンダーニュートラルなパイプラインを維持しながら Azure Monitor を活用できます。
  • メトリクス: Azure Monitor Workspace (Prometheus) に格納、PromQL でクエリ可能
  • ログ: Log Analytics Workspace に OpenTelemetry セマンティック規約で格納
  • トレース: Log Analytics Workspace に分散トレースとして格納
🔐
セキュア認証基盤
Microsoft Entra ID による認証と Data Collection Rules (DCR) によるアクセス制御で、キーレスかつセキュアなテレメトリパイプラインを構築。
  • マネージド ID / ワークロード ID / サービスプリンシパル対応
  • Monitoring Metrics Publisher ロールによる RBAC 制御
  • Azure Authentication Extension によるシームレスな認証
🔄
3 つのインジェストメカニズム
  • OpenTelemetry Collector (GA): あらゆる環境から Azure Monitor に直接送信
  • Azure Monitor Agent (プレビュー): Azure VM, VMSS, Arc サーバーからのインジェスト
  • AKS アドオン: AKS クラスターのコンテナアプリケーションからの収集
📐
Service Level Indicators (SLI)
CPU やメモリだけでなく、顧客体験を反映する複合インジケータを定義。アプリケーションのサービスレベルを定量的に計測できます。
  • 可用性、レイテンシ、エラー率を組み合わせた複合 SLI の定義
  • インフラメトリクスでは捉えられない顧客体験の可視化
  • チーム間で共通の品質指標を共有
🎯
Service Level Objectives (SLO)
SLI に対して目標値を設定し、達成状況を継続的に追跡。エラーバジェットによるリスク管理が可能になります。
  • SLO ターゲットの設定と達成状況の継続追跡
  • エラーバジェットの消費状況の可視化
  • SLO 違反リスクに対する早期警告アラート
🔗
Application Insights 統合
OTLP サポートを有効化した Application Insights リソースを作成可能。DCR リソース ID とエンドポイント URL が自動プロビジョニングされ、既存の分散トレースや障害分析機能で OTLP データを分析できます。

🛠️ Collector 設定例

OpenTelemetry Collector 設定 (YAML)

receivers:
  otlp:
    protocols:
      grpc:
        endpoint: localhost:4317
      http:
        endpoint: localhost:4318

processors:
  batch:

extensions:
  azure_auth:
    managed_identity: {}
    scopes:
      - https://monitor.azure.com/.default

exporters:
  otlphttp/azuremonitor:
    traces_endpoint: "https://<logs-dce-domain>/datacollectionRules/<dcr-immutable-id>/streams/Microsoft-OTLP-Traces/otlp/v1/traces"
    logs_endpoint: "https://<logs-dce-domain>/datacollectionRules/<dcr-immutable-id>/streams/Microsoft-OTLP-Logs/otlp/v1/logs"
    metrics_endpoint: "https://<metrics-dce-domain>/datacollectionRules/<dcr-immutable-id>/streams/Custom-Metrics-Otel/otlp/v1/metrics"
    auth:
      authenticator: azure_auth

service:
  extensions:
    - azure_auth
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlphttp/azuremonitor]
    metrics:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlphttp/azuremonitor]
    logs:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlphttp/azuremonitor]

💡 ユースケース

🌐 マルチクラウド統一オブザーバビリティ

AWS、GCP、Azure にまたがるマイクロサービスを OTel SDK で計装し、各環境の OpenTelemetry Collector から Azure Monitor に一元集約。ベンダー非依存の計装を維持しながら、Application Insights で end-to-end の分散トレースを横断的に分析。

📊 SLI/SLO によるサービス品質管理

可用性、レスポンスタイム (P95)、エラー率を組み合わせた複合 SLI を定義し、SLO 99.9% に対するエラーバジェット消費状況をダッシュボードで可視化。信頼性とイノベーション速度のバランスを定量的に管理。

🔧 既存 OTel Collector インフラの統合

既に Prometheus/Jaeger に送信している OpenTelemetry Collector に Azure Monitor を追加バックエンドとして設定。既存スタックを維持しながら Application Insights の障害分析・パフォーマンス分析を活用し、段階的な移行パスを確保。
