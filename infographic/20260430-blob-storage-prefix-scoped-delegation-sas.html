Azure Blob Storage GA Storage Security

User Delegation SAS のプレフィックススコープアクセス

ディレクトリ/プレフィックス単位でアクセスを制限する新しいスコープ (sr=d) により、最小権限の原則に基づいた安全なデータアクセス委任を実現

Release: 2026-04-30 Service: Azure Blob Storage Feature: Prefix-scoped access for User Delegation SAS

📋 Overview

Azure Blob Storage のユーザー委任 SAS (User Delegation SAS) におけるプレフィックススコープアクセスが、すべての Azure リージョンで一般提供 (GA) となった。従来、Blob Storage の SAS トークンはコンテナレベル (sr=c) と個別 Blob レベル (sr=b) の 2 つのスコープのみをサポートしていたが、本リリースにより新たにディレクトリ/プレフィックススコープ (sr=d) が利用可能となった。プレフィックススコープを使用すると、コンテナ内の特定のディレクトリ (パスプレフィックス) 配下のすべての Blob に対してアクセスを制限できる。

Before

  • SAS トークンのスコープがコンテナ全体か個別 Blob のいずれかに限定
  • テナントごと・ユーザーごとのパスプレフィックス分離が困難
  • コンテナスコープ SAS ではコンテナ内全 Blob にアクセス可能 (最小権限の原則に反する)
  • 個別 Blob スコープ SAS では多数の Blob に個別トークン発行が必要 (運用負荷大)

After

  • ディレクトリ/プレフィックススコープ (sr=d) で特定パス配下のみにアクセス制限
  • ユーザー委任 SAS との組み合わせでアカウントキー不要の安全な委任
  • signedDirectoryDepth (sdd) でアクセス可能な階層の深さを明示的に指定可能
  • Microsoft Entra ID ベースのプレフィックス単位アクセス委任が実現

プレフィックススコープ SAS のアクセスモデル。アプリケーションが Entra ID で認証後、User Delegation Key を取得し、sr=d パラメータにより特定のディレクトリプレフィックス配下の Blob のみにアクセスを制限する。

⚙ Key Features

📂

ディレクトリスコープ (sr=d) のサポート

signedResource パラメータに d を指定することで、特定のディレクトリ配下のリソースにアクセスを制限できる

  • コンテナ内の特定パスプレフィックス配下の全 Blob が対象
  • コンテナ全体への過剰なアクセス付与を回避
📏

ディレクトリ深度の指定 (sdd パラメータ)

ルートディレクトリからの深さを指定する signedDirectoryDepth パラメータ。sr=d の場合に必須

  • アクセス可能なディレクトリ階層の深さを明示的に制御
  • サブディレクトリへのアクセス範囲を精密に制限
🔑

ユーザー委任 SAS との組み合わせ

アカウントキーに依存しないセキュアなアクセス委任。Microsoft Entra ID の資格情報で署名

  • ストレージアカウントキーの漏洩リスクを完全排除
  • Entra ID の条件付きアクセスや MFA と連携可能
🌍

全 Azure リージョンでの一般提供

すべての Azure リージョンで GA として利用可能。追加費用なし

  • グローバルに統一されたセキュリティポリシーの適用が可能
  • 既存のストレージアカウントですぐに利用開始可能

🛠 Configuration

Azure CLI

bash Azure CLI - ロール割り当て & SAS 生成 
# 1. Storage Blob Delegator ロールの割り当て
az role assignment create \
  --assignee {principal-id} \
  --role "Storage Blob Delegator" \
  --scope /subscriptions/{sub-id}/resourceGroups/{rg}/providers/Microsoft.Storage/storageAccounts/{account}

# 2. プレフィックススコープのユーザー委任 SAS を生成
az storage blob generate-sas \
  --account-name {account} \
  --container-name mycontainer \
  --name "tenant-a/" \
  --as-user \
  --auth-mode login \
  --permissions rl \
  --expiry 2026-05-01T00:00:00Z \
  --resource-types d

💡 Use Cases

1

マルチテナント SaaS アプリケーション

SaaS プラットフォームで各テナントのデータを同一コンテナ内のパスプレフィックスで分離し、テナントごとに自分のデータのみにアクセスできる SAS を発行する。

✓ テナント間のデータ分離を SAS レベルで強制。コンテナを分けずに安全なマルチテナンシーを実現。
2

CI/CD パイプラインでのアーティファクト管理

プロジェクトごと・ビルドごとのパスプレフィックスにアーティファクトを保存し、各パイプラインに自身のプレフィックスのみへの書き込み権限を委任する。

✓ パイプラインが他プロジェクトのアーティファクトを誤って上書き・削除するリスクを排除。
3

データレイクの部門別アクセス制御

Data Lake Storage で部門別にディレクトリ構造を持つデータレイクにおいて、各部門のアナリストに自部門のデータへのみ読み取りアクセスを委任する。

✓ 部門間のデータアクセス境界を明確に設定し、コンプライアンス要件を満たす。

✨ Benefits

ビジネス面

  • 最小権限の原則の実現
  • マルチテナント環境での安全なデータ分離
  • 運用負荷の軽減

技術面

  • アカウントキー不要のセキュアなアクセス
  • 階層型ストレージとの親和性
  • 柔軟なパーミッション設定
  • 監査追跡性の向上

⚠ Limitations

⚠ API バージョン sv=2020-02-10 以降が必要
⚠ ストアドアクセスポリシー非対応
⚠ ユーザー委任キーの有効期間制限 (最大 7 日間)
⚠ コンテナの管理操作は不可
⚠ Blob Storage 専用 (Tables, Queues, Files には非対応)
