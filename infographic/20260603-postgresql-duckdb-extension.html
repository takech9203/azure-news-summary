GA 2026-06-03

Overview

Azure Database for PostgreSQL - Flexible Server において、DuckDB 拡張機能 (pg_duckdb) が一般提供 (GA) となった。PostgreSQL のトランザクション処理 (OLTP) 環境内に DuckDB の列指向ベクトル化分析エンジンを統合し、単一データベースインスタンス上で OLTP と OLAP (分析処理) の両方を実行できるようになる。

DuckDB は高速な分析クエリに特化したインメモリ列指向データベースエンジン。pg_duckdb 拡張機能を通じて PostgreSQL 内から直接利用可能。Microsoft Build 2026 にて発表され、Azure のデータプラットフォーム戦略における「データレイクハウス」アーキテクチャの一環として位置付けられる。

Key Points

  • DuckDB の列指向ベクトル化エンジンにより分析クエリが大幅に高速化
  • 同一 PostgreSQL インスタンス内で OLTP と OLAP を同時処理、データ複製不要
  • Azure Blob Storage 上の Parquet/CSV/JSON ファイルを直接クエリ可能
  • Apache Iceberg、Delta Lake 形式のテーブルに対応しデータレイクハウスを実現
  • MIT License で追加ライセンス費用なし

Before

  • PostgreSQL 上で大規模集計・分析クエリを実行するとパフォーマンスが低かった
  • 分析ワークロードのために別途 Azure Synapse Analytics やデータウェアハウスを構築し ETL パイプラインでデータを複製する必要があった
  • Parquet や Iceberg 等のデータレイク形式ファイルに PostgreSQL から直接アクセスする手段がなかった
  • OLTP と OLAP を単一システムで処理する HTAP アーキテクチャの構築が困難

After

  • DuckDB の列指向ベクトル化エンジンにより分析クエリが大幅に高速化
  • 同一 PostgreSQL インスタンス内で OLTP と OLAP を同時処理でき、データ複製が不要
  • read_parquet() 等の関数により Azure Blob Storage 上の Parquet/CSV/JSON ファイルを直接クエリ可能
  • Apache Iceberg、Delta Lake 形式のテーブルにも対応しデータレイクハウスアーキテクチャを実現

PostgreSQL のクエリプランナーを拡張し、分析クエリを自動的に DuckDB エンジンにルーティングする。トランザクション操作は従来通り PostgreSQL エンジンが処理し、両エンジンが同一インスタンス内でデータを共有する。外部ストレージ (Azure Blob Storage) 上の Parquet/Iceberg/Delta Lake ファイルも DuckDB エンジン経由で直接アクセス可能。

⚡

透過的クエリアクセラレーション

duckdb.force_execution=true の設定により分析クエリが自動的に DuckDB エンジンを経由して実行される。既存のクエリがそのまま高速化され、特別な SQL 構文の変更は不要。

💾

データレイク読み書き

Parquet、CSV、JSON、Apache Iceberg、Delta Lake 形式のファイルに対応。Azure Blob Storage、S3、GCS 上のファイルを直接クエリ可能。read_parquet() 関数でファイルをテーブルのように扱える。

🔗

PostgreSQL テーブルとの統合クエリ

ローカル PostgreSQL テーブルと外部データレイクファイルを単一クエリで JOIN 可能。データの移動やコピーなしにリアルタイム分析を実現。

🧩

DuckDB 内部拡張機能のサポート

duckdb.install_extension('iceberg') 等で DuckDB 自体の拡張機能をインストール可能。Iceberg テーブルのタイムトラベル (バージョンスナップショット) にも対応。

Code Example

拡張機能のインストールと分析クエリの実行

-- 拡張機能のインストール
CREATE EXTENSION pg_duckdb;

-- DuckDB エンジンの有効化
SET duckdb.force_execution = true;

-- 分析クエリ (自動的に DuckDB エンジンで処理)
SELECT
    date_trunc('month', order_date) AS month,
    COUNT(*) AS order_count,
    SUM(amount) AS total_amount
FROM orders
GROUP BY 1
ORDER BY 1;

-- Azure Blob Storage 上の Parquet ファイルをクエリ
SELECT * FROM read_parquet('az://<container>/<path>/data.parquet');

Use Cases

UC1: トランザクションデータのリアルタイム分析

EC サイトの注文データ (PostgreSQL に OLTP で格納) に対して、リアルタイムの売上集計・傾向分析を実行。別途分析基盤を構築せず、トランザクションデータベース上で直接集計クエリを高速実行。

効果: ETL 遅延がゼロ。データ鮮度がリアルタイムで分析コスト削減。

UC2: データレイクハウスクエリ

Azure Blob Storage に蓄積された過去の履歴データ (Parquet 形式) と現行の PostgreSQL テーブルを結合して分析。過去データをデータレイクに低コスト保存しつつ、現行データとシームレスに結合分析可能。

効果: データウェアハウスの構築が不要。アーキテクチャのシンプル化。

UC3: Iceberg テーブルによるタイムトラベル分析

データレイク上の Apache Iceberg テーブルに対して、特定時点のスナップショットをクエリし、データの変遷を追跡。PostgreSQL からデータレイクの Iceberg テーブルに直接アクセス。

効果: タイムトラベル機能を活用した監査・分析が可能。

分析ワークロードの特性上、メモリ最適化 SKU (E シリーズ) の選択が推奨される。DuckDB はインメモリ処理のため、利用可能メモリが多いほどパフォーマンスが向上する。

Constraints

  • DuckDB の高度な型 (STRUCT, MAP, UNION) は duckdb.query() コンテキスト内でのみ使用可能
  • duckdb.force_execution=true の設定が必要 (明示的な有効化)
  • 書き込み操作は引き続き PostgreSQL エンジンで処理
  • DuckDB エンジンはインメモリ処理のためメモリに応じてパフォーマンスが変動
