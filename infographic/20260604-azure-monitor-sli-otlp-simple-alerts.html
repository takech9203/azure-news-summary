概要

Microsoft Build 2026 にて、Azure Monitor のオブザーバビリティ機能に関する 3 つの重要なアップデートが一般提供 (GA) として発表された。SLI/SLO によるユーザー体験ベースの監視、OTLP ネイティブインジェストによるベンダーニュートラルなテレメトリパイプライン、Simple Log Alerts による個別ログイベントのリアルタイム監視を実現する。

3 つの GA 機能

  • SLI/SLO - 可用性・レイテンシー・スループットの観点からユーザー体験を定量的に測定
  • OTLP ネイティブインジェスト - OpenTelemetry Collector 経由で OTLP シグナルを直接 Azure Monitor に送信
  • Simple Log Alerts - 個々のログイベントに対するリアルタイムのイベント駆動型アラート

Before

  • インフラメトリクス (CPU、メモリ等) 依存の監視で、実際のユーザー体験を正確に把握できなかった
  • OpenTelemetry 計装アプリから Azure Monitor へ送信するには Microsoft 固有の SDK やエクスポーターが必要だった
  • 従来のログアラートは集約ベースの評価が必要で、個々のイベントに即座に反応するシンプルな設定が困難だった

After

  • SLI/SLO により可用性・レイテンシー・スループットのターゲットを定義し、SRE プラクティスを Azure 上で実践可能に
  • OpenTelemetry Collector から直接クラウドインジェストエンドポイントへ送信、ベンダーロックインなしの監視パイプラインを構築可能に
  • Simple Log Alerts により、各ログ行を個別に評価してイベント駆動型アラートを即座に発火可能に

1. Service Level Indicators (SLI) / Service Level Objectives (SLO)

アプリケーションのユーザー体験を定量的に測定し、SRE プラクティスを Azure 上で直接実践可能にする機能。

  • 可用性 (Availability): サービスが正常に応答している割合
  • レイテンシー (Latency): リクエスト処理の応答時間目標
  • スループット (Throughput): 単位時間あたりの処理量目標
  • エラーバジェットの概念を Azure 上で直接活用
📡

2. OTLP ネイティブインジェスト

OpenTelemetry Collector から直接 Azure Monitor へシグナルを送信するベンダーニュートラルなパイプライン。

  • メトリクス: Azure Monitor Workspace (Prometheus データストア)、PromQL でクエリ可能
  • ログ: Log Analytics Workspace に OpenTelemetry セマンティック規約で保存
  • トレース: Log Analytics Workspace に分散トレースとして保存
  • 認証: Microsoft Entra ID (マネージド ID / サービスプリンシパル / ワークロード ID)
🔔

3. Simple Log Alerts

ログアラートの作成・管理を簡素化し、個々のログイベントに対するリアルタイム監視を実現する機能。

  • 各ログ行を個別に評価 (非集約型)
  • Analytics ログと Basic ログの両テーブルプランに対応
  • Transformation KQL 言語に基づくシンプルなクエリ構文
  • 1 分間あたりの条件一致回数でトリガー閾値を設定可能

設定例: OTLP Collector YAML

receivers:
  otlp:
    protocols:
      grpc:
        endpoint: localhost:4317
      http:
        endpoint: localhost:4318

processors:
  batch:

extensions:
  azure_auth:
    managed_identity: {}
    scopes:
      - https://monitor.azure.com/.default

exporters:
  otlphttp/azuremonitor:
    traces_endpoint: "https://<logs-dce-domain>/dataCollectionRules/<dcr-immutable-id>/streams/Microsoft-OTLP-Traces/otlp/v1/traces"
    logs_endpoint: "https://<logs-dce-domain>/dataCollectionRules/<dcr-immutable-id>/streams/Microsoft-OTLP-Logs/otlp/v1/logs"
    metrics_endpoint: "https://<metrics-dce-domain>/dataCollectionRules/<dcr-immutable-id>/streams/Custom-Metrics-Otel/otlp/v1/metrics"
    auth:
      authenticator: azure_auth

service:
  extensions:
    - azure_auth
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlphttp/azuremonitor]
    metrics:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlphttp/azuremonitor]
    logs:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlphttp/azuremonitor]

Use Cases

UC1: SRE チームによる SLO ベースの運用

E コマースプラットフォームで可用性 99.9%、P95 レイテンシー 200ms 以下の SLO を定義。エラーバジェットの消費状況をリアルタイムで可視化し、バジェット超過時にデプロイメント凍結を自動実行する。

-- インフラメトリクスではなく顧客体験に基づいた運用判断が可能に

UC2: マルチクラウド環境の統一オブザーバビリティ

AWS と Azure のハイブリッド環境で OpenTelemetry SDK で計装。OpenTelemetry Collector を中心としたパイプラインにより、マルチクラウド環境のテレメトリを Azure Monitor に集約。ベンダーロックインなしの統一監視基盤を構築する。

-- 単一の計装標準 (OpenTelemetry) でマルチクラウド環境を統一的に監視

UC3: バックアップジョブ失敗の即時検知

Azure Backup のジョブ失敗ログを Simple Log Alerts で監視。集約ベースの評価を待たず、各ログイベントを即座に評価してアラートを発火する。1 分間の閾値設定でノイズを制御しつつリアルタイム検知を実現。

-- 従来の集約型アラートでは検知に遅延が発生していた問題を解消

制約事項

  • OTLP インジェストの OpenTelemetry Collector パスのみ GA。AMA および AKS パスはプレビュー
  • OTLP メトリクスは Delta テンポラリティが必須
  • Simple Log Alerts はディメンションによる分割やミュートアクションに非対応
  • OpenTelemetry Collector の OSS コンポーネントに対する Azure サポートはコミュニティチャネル経由のみ
